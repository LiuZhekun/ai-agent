> 本文详细说明 MySQL / Redis / Milvus 的部署方式，包含一键部署、逐步手动部署、验证方法和常见问题排查。

## 前置条件

| 条件 | 说明 |
|------|------|

## 方式一：一键部署（推荐）

适合快速启动，脚本自动完成 Docker 安装检测、容器拉起和数据库初始化。

### Linux / macOS

```bash
cd deploy
chmod +x install-all.sh
./install-all.sh
```

### Windows PowerShell

Windows 下无法直接执行 `.sh` 脚本，使用 Docker Compose 命令替代：

```powershell
cd deploy
docker compose --env-file .env.example up -d mysql redis milvus
```

### 脚本做了什么？

| 步骤 | 说明 |
|------|------|
| 1. 检测 Docker | 未安装则自动安装（支持 Ubuntu/CentOS） |
| 2. 检测 Compose | 确认 Docker Compose Plugin 可用 |
| 3. 生成 .env | 若无 `.env` 文件，自动从 `.env.example` 复制 |
| 4. 启动容器 | 拉取镜像并启动 MySQL + Redis + Milvus |
| 5. 等待就绪 | 通过健康检查等待所有服务变为 healthy |
| 6. 验证数据库 | 检查初始化表是否已创建 |
| 7. 输出地址 | 打印各服务的访问地址和常用命令 |

### 一键部署参数

| 参数 | 说明 |
|------|------|
| （无参数） | 仅启动中间件（MySQL + Redis + Milvus），适合本地开发 |
| `--full` | 启动全部服务，包含 Nginx + Backend（需先构建后端镜像） |
| `--init-db` | 强制重新执行数据库初始化脚本（**慎用，会重建表**） |
| `--help` | 显示帮助信息 |

## 方式二：手动逐步部署

如果你需要更精细的控制，可以按以下步骤手动操作。

### 步骤 1：准备环境文件

```bash
cd deploy
cp .env.example .env
```

如需修改端口或密码，编辑 `.env` 文件：

```properties
# .env 可配置项
MYSQL_ROOT_PASSWORD=root      # MySQL root 密码
MYSQL_DATABASE=ai_agent       # 数据库名
MYSQL_PORT=3306               # MySQL 端口
REDIS_PORT=6379               # Redis 端口
MILVUS_PORT=19530             # Milvus 端口
MILVUS_METRICS_PORT=9091      # Milvus 健康检查端口
```

### 步骤 2：拉取镜像

```bash
docker compose --env-file .env pull mysql redis milvus
```

> 首次拉取约需 5-15 分钟，视网络速度而定。如果超时，请配置 Docker 镜像加速器。

### 步骤 3：启动服务

```bash
docker compose --env-file .env up -d mysql redis milvus
```

### 步骤 4：验证服务状态

```bash
docker compose ps
```

预期输出（3 个容器均为 `Up (healthy)`）：

```
NAME                STATUS              PORTS
ai-agent-mysql      Up (healthy)        0.0.0.0:3306->3306/tcp
ai-agent-redis      Up (healthy)        0.0.0.0:6379->6379/tcp
ai-agent-milvus     Up (healthy)        0.0.0.0:19530->19530/tcp
```

> **注意**：Milvus 首次启动较慢（约 30-90 秒），需等待健康检查通过。
> 可通过 `docker compose logs -f milvus` 观察启动日志。

### 步骤 5：验证数据库初始化

MySQL 首次启动时会自动执行 `deploy/init-db/` 下的 SQL 脚本，创建业务表和示例数据。

```bash
# 检查表是否已创建
docker compose exec mysql mysql -uroot -proot ai_agent -e "SHOW TABLES;"
```

预期输出：

```
+--------------------+
| Tables_in_ai_agent |
+--------------------+
| sys_department     |
| sys_dict           |
| sys_user           |
+--------------------+
```

```bash
# 检查示例数据
docker compose exec mysql mysql -uroot -proot ai_agent -e "SELECT COUNT(*) FROM sys_user;"
```

预期输出：`11`（11 条用户记录）。

## 方式三：单独部署某个中间件

每个中间件在 `deploy/` 子目录下有独立的 `docker-compose.yml` 和 `install.sh`，可单独部署。

```bash
# 仅启动 MySQL
cd deploy/mysql && chmod +x install.sh && ./install.sh

# 仅启动 Redis
cd deploy/redis && chmod +x install.sh && ./install.sh

# 仅启动 Milvus
cd deploy/milvus && chmod +x install.sh && ./install.sh
```

> 单独部署 MySQL 时不会自动执行 `init-db` 脚本，需手动导入：
>
> ```bash
> mysql -h127.0.0.1 -P3306 -uroot -proot ai_agent < deploy/init-db/schema.sql
> mysql -h127.0.0.1 -P3306 -uroot -proot ai_agent < deploy/init-db/data.sql
> ```

## 方式四：离线部署（无公网环境）

### 在联网机器上导出镜像

```bash
cd deploy/offline
chmod +x export-images.sh
./export-images.sh
```

产物：`deploy/offline/images/ai-agent-images.tar`（约 3-5 GB）

### 传输到目标机器

将 `ai-agent-images.tar` 和整个 `deploy/` 目录拷贝到目标机器。

### 在目标机器上导入并启动

```bash
cd deploy/offline
chmod +x import-images.sh
./import-images.sh

cd ..
chmod +x install-all.sh
./install-all.sh
```

## 各中间件说明

### MySQL 8.0

| 项目 | 说明 |
|------|------|
| 镜像 | `mysql:8.0` |
| 默认端口 | 3306 |
| 默认账号 | root / root |
| 默认库名 | ai_agent |
| 字符集 | utf8mb4（通过 `my.cnf` 和启动参数双重保证） |
| 数据持久化 | 命名卷 `ai-agent-mysql-data`，容器删除后数据仍保留 |
| 初始化脚本 | `deploy/init-db/schema.sql`（建表）+ `data.sql`（示例数据） |

### Redis 7

| 项目 | 说明 |
|------|------|
| 镜像 | `redis:7` |
| 默认端口 | 6379 |
| 默认密码 | 无（生产环境务必设置） |
| 持久化 | AOF 模式（通过 `redis.conf` 中 `appendonly yes` 配置） |
| 数据持久化 | 命名卷 `ai-agent-redis-data` |

### Milvus 2.4

| 项目 | 说明 |
|------|------|
| 镜像 | `milvusdb/milvus:v2.4.13` |
| 默认端口 | 19530（gRPC）+ 9091（健康检查/指标） |
| 运行模式 | Standalone（嵌入式 etcd + 本地存储，无需额外容器） |
| 数据持久化 | 命名卷 `ai-agent-milvus-data` |
| 启动耗时 | 首次约 30-90 秒，请耐心等待 |

## 数据持久化说明

所有中间件数据通过 Docker 命名卷持久化：

| 卷名 | 用途 |
|------|------|
| `ai-agent-mysql-data` | MySQL 数据文件 |
| `ai-agent-redis-data` | Redis AOF 持久化文件 |
| `ai-agent-milvus-data` | Milvus 向量数据和 etcd 元数据 |

**重要**：
- `docker compose stop` → 停止容器，数据保留
- `docker compose down` → 删除容器，数据保留（卷仍在）
- `docker compose down -v` → 删除容器 **并清除所有数据**（慎用）
- `./uninstall-all.sh` → 删除容器，数据保留
- `./uninstall-all.sh --purge-volumes` → 删除容器 **并清除所有数据**

查看当前数据卷：

```bash
docker volume ls | grep ai-agent
```

## 建议虚拟机规格

| 场景 | CPU | 内存 | 磁盘 | 网络 |
|------|-----|------|------|------|
| 单机演示 | 4 vCPU | 8 GB | 80 GB SSD | 100 Mbps |
| 多人联调 | 8 vCPU | 16 GB | 150 GB SSD | 1 Gbps |
| 生产环境 | 16+ vCPU | 32+ GB | 300+ GB NVMe SSD | 1+ Gbps |

**生产部署建议**：
- MySQL 与 Milvus 使用独立存储卷，避免容器重建丢数据
- 日志接入集中式采集（ELK / Loki），避免磁盘被打满
- MySQL、Milvus 与应用层拆分部署，降低资源争用

## 常用运维命令

```bash
# 查看所有容器状态
docker compose ps

# 查看某个服务的日志（实时跟踪）
docker compose logs -f mysql
docker compose logs -f redis
docker compose logs -f milvus

# 重启某个服务
docker compose restart mysql

# 停止全部服务（保留数据）

# 启动全部服务
docker compose start

# 进入 MySQL 命令行
docker compose exec mysql mysql -uroot -proot ai_agent

# 进入 Redis 命令行
docker compose exec redis redis-cli

# 查看数据卷占用
docker system df -v | grep ai-agent
```

## 常见问题排查

| 问题 | 排查方向 | 解决方案 |
|------|---------|---------|
| 镜像拉取超时 | 网络不通或无加速器 | 配置 Docker 镜像加速器（搜索"Docker 镜像加速"） |
| 端口被占用 | 本机已有 MySQL/Redis 等进程占用端口 | 修改 `.env` 中的端口号，或停止占用进程 |
| MySQL 连接被拒绝 | 容器尚未启动完成 | `docker compose ps` 确认状态为 healthy，首次启动需等 30 秒 |
| MySQL 表不存在 | 初始化脚本未执行 | 方法1: `./install-all.sh --init-db`；方法2: 手动执行 SQL |
| MySQL 字符乱码 | 字符集配置缺失 | 确认 `my.cnf` 已挂载，容器启动命令含 `--character-set-server=utf8mb4` |
| Redis 数据重启后丢失 | 未开启持久化 | 确认 `redis.conf` 中 `appendonly yes` 且卷已挂载 |
| Milvus 启动超慢 | 首次初始化 etcd 和索引 | 正常现象，需 30-90 秒，通过 `docker compose logs -f milvus` 观察 |
| Milvus 健康检查一直失败 | 内存不足 | Milvus 建议至少 4 GB 可用内存，检查 `docker stats` |
| Milvus 连接 19530 失败 | 容器未完全就绪 | 确认 `http://127.0.0.1:9091/healthz` 返回 `OK` |
| `docker compose` 命令找不到 | Docker Compose V2 未安装 | 安装 `docker-compose-plugin`，或升级 Docker Desktop |
| 权限不足 | Linux 非 root 用户 | 将用户加入 docker 组：`sudo usermod -aG docker $USER`，然后重新登录 |
| `install-all.sh` 执行报错 | 文件无执行权限 | `chmod +x install-all.sh` |
| 数据被意外清除 | 执行了 `docker compose down -v` | `-v` 参数会删除数据卷，日常操作请用 `stop` 或不带 `-v` 的 `down` |

## 完整操作速查

```bash
# 1. 首次部署（一键）
cd deploy
chmod +x install-all.sh
./install-all.sh

# 2. Windows 首次部署
cd deploy
docker compose --env-file .env.example up -d mysql redis milvus

# 3. 验证状态
docker compose ps

# 4. 日常停止/启动
docker compose stop
docker compose start

# 5. 完全卸载（保留数据）
./uninstall-all.sh

# 6. 完全卸载（清除数据，慎用）
./uninstall-all.sh --purge-volumes

# 7. 强制重新初始化数据库
./install-all.sh --init-db

# 8. 完整部署（含后端+Nginx）
cd deploy/backend && ./build-and-deploy.sh
cd .. && ./install-all.sh --full
```
