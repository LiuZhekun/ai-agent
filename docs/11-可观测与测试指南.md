# 可观测与测试指南

## 1. 概述

在 AI Agent 系统中，一次用户请求往往经历「意图识别 → 澄清 → 工具调用 → SQL 执行 → 字段翻译 → 流式响应」等多个异步环节。每个环节都可能失败、超时或产生超出预期的资源消耗。如果缺乏有效的观测手段，问题将极难定位。

可观测性（Observability）的三大支柱——**指标（Metrics）**、**日志（Logging）**、**追踪（Tracing）**——在 Agent 系统中的角色如下：

| 支柱 | 在 Agent 系统中的作用 | 核心组件 |
|------|----------------------|---------|
| **指标** | 量化 Token 消耗、工具调用延迟、SQL 拒绝率等关键信号，用于告警与容量规划 | `AgentMetrics` + Micrometer |
| **日志** | 结构化记录每次工具调用、SQL 审计、澄清回合的完整上下文，用于事后分析 | `AgentAuditLogger` + Logback |
| **追踪** | 以 `traceId` 串联同一请求在 SSE 事件、审计日志、执行轨迹中的全部记录 | `ExecutionTrace` + `AgentEvent` |

三者协同工作：指标发现异常 → 追踪定位请求 → 日志还原现场。

## 2. Micrometer 指标

### 2.1 自动装配机制

指标采集通过 `AgentMetricsAutoConfiguration` 条件装配：

```java
@AutoConfiguration
@ConditionalOnClass(MeterRegistry.class)
public class AgentMetricsAutoConfiguration {

    @Bean
    @ConditionalOnBean(MeterRegistry.class)
    public AgentMetrics agentMetrics(MeterRegistry meterRegistry) {
        return new AgentMetrics(meterRegistry);
    }
}
```

装配条件：
1. Classpath 中存在 Micrometer 的 `MeterRegistry` 类（即引入了 `spring-boot-starter-actuator`）。
2. Spring 容器中已注册 `MeterRegistry` Bean。

如果项目不依赖 Actuator，整个指标体系不会激活，不影响核心业务功能。

### 2.2 Token 消耗指标

| 指标名称 | 类型 | 标签 | 含义 |
|---------|------|------|------|
| `agent.token.consumed` | Counter | 无 | 累计消耗的 Token 总量（promptTokens + completionTokens） |

调用方式：

```java
agentMetrics.recordTokenUsage(promptTokens, completionTokens);
```

**使用场景**：每次 LLM 调用完成后记录。可用于：
- 按时间维度统计 Token 用量趋势
- 设置用量告警阈值，防止成本失控
- 结合 Grafana 面板展示日/周/月消耗曲线

### 2.3 工具调用指标

| 指标名称 | 类型 | 标签 | 含义 |
|---------|------|------|------|
| `agent.tool.calls` | Counter | `tool.name`, `success` | 每次工具调用计数，区分工具名称和成功/失败 |
| `agent.tool.duration` | Timer | `tool.name` | 工具调用耗时分布（单位：毫秒） |

调用方式：

```java
agentMetrics.recordToolCall("queryUser", true, 128L);
```

**通过标签组合可以计算**：
- **调用次数**：`agent.tool.calls` 按 `tool.name` 分组
- **成功率**：`agent.tool.calls{success=true}` / `agent.tool.calls` × 100%
- **耗时分布**：`agent.tool.duration` 的 P50/P95/P99 分位数
- **慢调用发现**：`agent.tool.duration` max 值突增

### 2.4 SQL 相关指标

| 指标名称 | 类型 | 标签 | 含义 |
|---------|------|------|------|
| `agent.sql.rejected` | Counter | `reason` | SQL 被安全策略拒绝的次数，按拒绝原因分类 |

调用方式：

```java
agentMetrics.recordSqlRejection("DDL_NOT_ALLOWED");
```

`reason` 标签的典型取值：
- `DDL_NOT_ALLOWED` — 检测到 DDL 语句（CREATE / DROP / ALTER）
- `DML_NOT_ALLOWED` — 检测到写操作（INSERT / UPDATE / DELETE）
- `TABLE_NOT_IN_WHITELIST` — 访问了未授权的表
- `SYNTAX_ERROR` — SQL 语法不合法
- `RATE_LIMITED` — 超过频率限制

### 2.5 会话指标

| 指标名称 | 类型 | 标签 | 含义 |
|---------|------|------|------|
| `agent.session.duration` | Timer | `session.id` | 单次会话从建立到结束的总耗时 |
| `agent.session.active` | Gauge | 无 | 当前活跃会话数 |

调用方式：

```java
agentMetrics.recordSessionDuration("sess-abc-123", 4500L);
agentMetrics.setActiveSessions(12);
```

**使用场景**：
- 监控会话耗时分布，识别异常长会话
- 追踪并发会话数，评估系统负载
- 设置活跃会话数告警阈值

### 2.6 MetricsConstants 常量一览

`MetricsConstants` 类集中定义了所有指标名称和标签键，避免硬编码字符串散落各处：

```java
public final class MetricsConstants {

    // -------- 指标名称 --------
    public static final String AGENT_TOKEN_CONSUMED    = "agent.token.consumed";
    public static final String AGENT_TOOL_CALLS        = "agent.tool.calls";
    public static final String AGENT_TOOL_SUCCESS_RATE = "agent.tool.success.rate";
    public static final String AGENT_TOOL_DURATION     = "agent.tool.duration";
    public static final String AGENT_SQL_REJECTED      = "agent.sql.rejected";
    public static final String AGENT_SESSION_DURATION  = "agent.session.duration";
    public static final String AGENT_SESSION_ACTIVE    = "agent.session.active";

    // -------- 标签键 --------
    public static final String TAG_TOOL_NAME   = "tool.name";
    public static final String TAG_TOOL_GROUP  = "tool.group";
    public static final String TAG_SESSION_ID  = "session.id";
    public static final String TAG_ERROR_TYPE  = "error.type";
}
```

> **开发规范**：新增指标时，先在 `MetricsConstants` 中添加常量，再在 `AgentMetrics` 中引用，禁止直接使用魔法字符串。

## 3. Actuator 端点配置

### 3.1 添加依赖

确保 `pom.xml` 中包含 Actuator 依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

### 3.2 暴露端点

在 `application.yml` 中配置：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus
  endpoint:
    health:
      show-details: when-authorized
  metrics:
    tags:
      application: ${spring.application.name}
```

### 3.3 常用端点

| 端点 | URL | 用途 |
|------|-----|------|
| 健康检查 | `GET /actuator/health` | 检查应用及依赖（DB、Redis）的健康状态 |
| 指标列表 | `GET /actuator/metrics` | 列出所有已注册的指标名称 |
| 指标详情 | `GET /actuator/metrics/{metricName}` | 查看指定指标的当前值 |
| Prometheus | `GET /actuator/prometheus` | 以 Prometheus 格式导出所有指标 |

### 3.4 查看 Agent 指标示例

```bash
# 查看 Token 消耗总量
curl http://localhost:8080/actuator/metrics/agent.token.consumed

# 查看特定工具的调用次数
curl http://localhost:8080/actuator/metrics/agent.tool.calls?tag=tool.name:queryUser

# 查看工具调用耗时的 P95
curl http://localhost:8080/actuator/metrics/agent.tool.duration?tag=tool.name:queryUser

# 查看 SQL 拒绝计数（按原因）
curl http://localhost:8080/actuator/metrics/agent.sql.rejected?tag=reason:DDL_NOT_ALLOWED

# 查看当前活跃会话数
curl http://localhost:8080/actuator/metrics/agent.session.active
```

### 3.5 接入 Prometheus + Grafana（可选）

添加 Prometheus 注册表依赖：

```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

Prometheus 抓取配置：

```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'ai-agent'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['localhost:8080']
```

推荐的 Grafana 面板指标：
- Token 消耗速率：`rate(agent_token_consumed_total[5m])`
- 工具调用成功率：`rate(agent_tool_calls_total{success="true"}[5m]) / rate(agent_tool_calls_total[5m])`
- 工具调用 P95 延迟：`histogram_quantile(0.95, rate(agent_tool_duration_seconds_bucket[5m]))`
- SQL 拒绝速率：`rate(agent_sql_rejected_total[5m])`
- 活跃会话数：`agent_session_active`

## 4. 结构化审计日志

### 4.1 AgentAuditLogger 概述

`AgentAuditLogger` 是审计与合规能力的基础组件，以 JSON 格式输出结构化日志，便于后续通过 ELK/Loki 等工具检索和分析。

它使用独立的 Logger 名称 `agent-audit`，与业务日志分离：

```java
private static final Logger LOG = LoggerFactory.getLogger("agent-audit");
```

### 4.2 审计事件类型

#### 4.2.1 工具调用事件 — `tool_call`

```java
auditLogger.logToolCall(traceId, sessionId, userId, toolName, input, output, durationMs, success);
```

输出示例：

```json
{
  "event": "tool_call",
  "traceId": "trace-abc-123",
  "sessionId": "sess-xyz-789",
  "timestamp": "2026-03-11T08:30:00.123Z",
  "userId": "user-001",
  "toolName": "queryUser",
  "input": {"name": "张三"},
  "output": [{"id": 1, "name": "张三", "dept": "研发部"}],
  "durationMs": 128,
  "success": true
}
```

#### 4.2.2 澄清回合事件 — `clarification`

```java
auditLogger.logClarification(traceId, sessionId, missingSlots, round);
```

输出示例：

```json
{
  "event": "clarification",
  "traceId": "trace-abc-123",
  "sessionId": "sess-xyz-789",
  "timestamp": "2026-03-11T08:30:01.456Z",
  "missingSlots": ["department", "startDate"],
  "round": 1
}
```

#### 4.2.3 SQL 执行事件 — `sql_execution`

```java
auditLogger.logSqlExecution(traceId, sessionId, sql, allowed, rejectReason);
```

输出示例（放行）：

```json
{
  "event": "sql_execution",
  "traceId": "trace-abc-123",
  "sessionId": "sess-xyz-789",
  "timestamp": "2026-03-11T08:30:02.789Z",
  "sql": "SELECT name, dept FROM sys_user WHERE status = 1",
  "allowed": true,
  "rejectReason": null
}
```

输出示例（拒绝）：

```json
{
  "event": "sql_execution",
  "traceId": "trace-abc-123",
  "sessionId": "sess-xyz-789",
  "timestamp": "2026-03-11T08:30:02.789Z",
  "sql": "DROP TABLE sys_user",
  "allowed": false,
  "rejectReason": "DDL_NOT_ALLOWED"
}
```

### 4.3 公共字段说明

所有审计事件都包含以下基础字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `event` | String | 事件类型标识（`tool_call` / `clarification` / `sql_execution`） |
| `traceId` | String | 全链路追踪 ID，串联同一请求的所有事件 |
| `sessionId` | String | 会话 ID，标识用户的对话上下文 |
| `timestamp` | String | ISO 8601 格式的时间戳 |

### 4.4 使用建议

`AgentAuditLogger` 默认已注册为 Spring Bean（`@Component`），但**不会自动接入每条业务链路**。需要审计时，在对应流程中显式注入并调用：

```java
@Service
public class MyToolService {

    private final AgentAuditLogger auditLogger;

    public MyToolService(AgentAuditLogger auditLogger) {
        this.auditLogger = auditLogger;
    }

    public Object execute(String traceId, String sessionId, Object input) {
        long start = System.currentTimeMillis();
        try {
            Object result = doExecute(input);
            auditLogger.logToolCall(traceId, sessionId, "currentUser",
                    "myTool", input, result,
                    System.currentTimeMillis() - start, true);
            return result;
        } catch (Exception e) {
            auditLogger.logToolCall(traceId, sessionId, "currentUser",
                    "myTool", input, e.getMessage(),
                    System.currentTimeMillis() - start, false);
            throw e;
        }
    }
}
```

## 5. Logback 配置

### 5.1 demo 项目中的 logback-spring.xml

项目的 `ai-agent-demo` 模块提供了推荐的日志配置，其核心设计是将**审计日志与业务日志分离**：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- 控制台输出：业务日志 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [%thread] %logger - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 滚动文件输出：审计日志 -->
    <appender name="AUDIT_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/agent-audit.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/agent-audit.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>7</maxHistory>
        </rollingPolicy>
        <encoder>
            <!-- 仅输出消息体，因为 AgentAuditLogger 已输出完整 JSON -->
            <pattern>%msg%n</pattern>
        </encoder>
    </appender>

    <!-- 审计 Logger：独立文件，不传播到 root -->
    <logger name="agent-audit" level="INFO" additivity="false">
        <appender-ref ref="AUDIT_FILE"/>
    </logger>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
    </root>
</configuration>
```

### 5.2 配置要点

| 配置项 | 说明 |
|--------|------|
| Logger 名称 `agent-audit` | 与 `AgentAuditLogger` 中的 `LoggerFactory.getLogger("agent-audit")` 对应 |
| `additivity="false"` | 审计日志不会同时输出到控制台，避免敏感信息泄露 |
| Encoder pattern `%msg%n` | 仅输出 JSON 消息体，因为时间戳等信息已包含在 JSON 中 |
| `maxHistory=7` | 保留最近 7 天的审计日志文件 |
| 滚动策略 | 按天滚动，文件名格式 `agent-audit.2026-03-11.log` |

### 5.3 扩展建议

**如果需要将审计日志推送到 ELK**，添加 Logstash Appender：

```xml
<appender name="AUDIT_LOGSTASH" class="net.logstash.logback.appender.LogstashTcpSocketAppender">
    <destination>logstash-host:5044</destination>
    <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
</appender>

<logger name="agent-audit" level="INFO" additivity="false">
    <appender-ref ref="AUDIT_FILE"/>
    <appender-ref ref="AUDIT_LOGSTASH"/>
</logger>
```

**如果需要同时在控制台看到审计日志**（仅限开发环境），使用 Spring Profile：

```xml
<springProfile name="dev">
    <logger name="agent-audit" level="INFO" additivity="false">
        <appender-ref ref="AUDIT_FILE"/>
        <appender-ref ref="CONSOLE"/>
    </logger>
</springProfile>

<springProfile name="!dev">
    <logger name="agent-audit" level="INFO" additivity="false">
        <appender-ref ref="AUDIT_FILE"/>
    </logger>
</springProfile>
```

## 6. 全链路追踪

### 6.1 traceId 的生成与传递

`traceId` 是贯穿单次用户请求全生命周期的唯一标识。它在以下三个维度中保持一致：

```
用户请求
  │
  ├── SSE 事件 (AgentEvent.traceId)
  │     ├── THINKING 事件 ─── traceId: "trace-abc-123"
  │     ├── TOOL_CALL 事件 ── traceId: "trace-abc-123"
  │     └── ANSWER 事件 ───── traceId: "trace-abc-123"
  │
  ├── 执行轨迹 (ExecutionTrace.traceId)
  │     ├── queryUser ──────── traceId: "trace-abc-123"
  │     └── addUser ────────── traceId: "trace-abc-123"
  │
  └── 审计日志 (AgentAuditLogger)
        ├── tool_call ──────── traceId: "trace-abc-123"
        ├── sql_execution ──── traceId: "trace-abc-123"
        └── clarification ──── traceId: "trace-abc-123"
```

### 6.2 SSE 事件中的 traceId

`AgentEvent` 模型定义了 `traceId` 字段，会随 SSE 推送到前端：

```java
public class AgentEvent implements Serializable {
    private String eventId;
    private String sessionId;
    private AgentEventType type;
    private Instant timestamp;
    private Object payload;
    private String traceId;    // 全链路追踪 ID
}
```

前端收到 SSE 事件后，可以提取 `traceId` 用于：
- 问题反馈时提供追踪线索
- 前端开发者工具中展示完整调用链
- 关联后端审计日志

### 6.3 执行轨迹中的 traceId

`ExecutionTrace` 记录了每次工具调用的完整生命周期，包含 `traceId`：

```java
public class ExecutionTrace implements Serializable {
    private String traceId;
    private String toolName;
    private String toolGroup;
    private Map<String, Object> input;
    private Object output;
    private TraceStatus status;   // STARTED / SUCCESS / FAILED
    private Instant startTime;
    private long durationMs;
    private String errorMessage;
}
```

执行轨迹由 `TraceAdvisor` 在工具调用前后创建和完善，最终附加到 `AgentSession.getExecutionTraces()` 中。

### 6.4 日志检索示例

当用户报告问题时，可通过 `traceId` 快速还原完整调用链：

```bash
# 在审计日志文件中检索某次请求的所有事件
grep "trace-abc-123" logs/agent-audit.log | jq .

# 如果接入了 ELK，使用 Kibana 查询
# traceId: "trace-abc-123" AND event: "sql_execution"
```

### 6.5 与 Spring Cloud Sleuth / Micrometer Tracing 集成

如果项目已接入分布式追踪（如 Zipkin / Jaeger），建议将 Agent 的 `traceId` 与 Span 关联：

```java
import io.micrometer.tracing.Tracer;

String traceId = tracer.currentSpan().context().traceId();
agentEvent.setTraceId(traceId);
```

这样可以在分布式追踪系统中看到 Agent 的调用链路。

## 7. 测试分层策略

### 7.1 测试金字塔

```
        ╱  端到端测试  ╲           ← 少量、慢、验证完整用户流程
       ╱   集成测试    ╲          ← 适量、验证模块间协作
      ╱    单元测试     ╲         ← 大量、快速、验证核心逻辑
     ╱───────────────────╲
```

### 7.2 单元测试

**目标**：验证核心算法和独立组件的正确性，不依赖外部基础设施。

| 被测组件 | 所属模块 | 验证重点 |
|---------|---------|---------|
| Token 裁剪器 (`TokenBudgetTrimmer`) | ai-agent-core | 超出预算时是否正确截断历史消息 |
| 摘要压缩器 (`SummaryCompressor`) | ai-agent-core | 历史消息压缩是否保留关键信息 |
| 澄清管理器 (`ClarificationManager`) | ai-agent-core | 缺失参数检测与多轮澄清状态管理 |
| 工具回调装饰器 (`ToolCallbackDecorator`) | ai-agent-core | 超时/重试/限流等装饰能力 |
| SQL 校验器 (`SqlValidator`) | ai-agent-knowledge | DDL/DML 拦截、白名单校验 |
| SQL 频率限制器 (`SqlRateLimiter`) | ai-agent-knowledge | 超频拒绝逻辑 |
| 字典翻译器 (`DictTranslator`) | ai-agent-translator | 多结果策略、未找到策略 |
| 实体引用翻译器 (`EntityRefTranslator`) | ai-agent-translator | 表白名单校验、翻译策略 |
| 翻译策略 (`TranslationPolicy`) | ai-agent-translator | 默认策略值 |
| 翻译装饰器 (`TranslateToolCallbackDecorator`) | ai-agent-translator | 工具调用前的翻译注入 |

### 7.3 集成测试

**目标**：验证模块间协作和外部依赖交互的正确性。

| 测试场景 | 验证重点 |
|---------|---------|
| 数据库工具调用 | Spring 上下文启动 → `@AgentTool` 扫描注册 → 查询执行 → 翻译生效 |
| SSE 流 | 建立连接 → 接收 THINKING/TOOL_CALL/ANSWER 事件 → 连接关闭 |
| Schema 发现 | 连接真实/嵌入式数据库 → 自动发现表结构 → 缓存验证 |
| 向量同步 | 数据变更 → 向量化 → 写入向量数据库 → 相似度检索 |

### 7.4 端到端测试

**目标**：从用户角度验证完整业务流程。

推荐场景：
- 用户提问 → Agent 返回准确答案（含工具调用链）
- 参数缺失 → Agent 发起澄清 → 用户补充 → Agent 完成调用
- 危险 SQL → Agent 拒绝执行并给出解释
- 并发会话 → 各会话互不干扰

## 8. 测试编写指南

### 8.1 Mock 策略

Agent 系统依赖多种外部组件，测试时需要合理 Mock：

| 依赖 | Mock 方式 | 说明 |
|------|----------|------|
| `ChatClient` (LLM) | Mockito `mock()` | 模拟 LLM 返回固定文本或工具调用指令 |
| `DataSource` / `JdbcTemplate` | Mockito `mock()` 或 H2 内嵌数据库 | 单元测试用 Mock，集成测试用 H2 |
| `Redis` | Embedded Redis 或 Mockito | 推荐使用 `spring-boot-starter-data-redis` 的 Test 支持 |
| `MeterRegistry` | `SimpleMeterRegistry` | Micrometer 提供的内存实现，用于验证指标注册 |
| 向量数据库 | Mockito `mock()` | Mock `VectorStore` 接口 |

### 8.2 翻译器测试示例详解

以 `DictTranslatorTest` 为例，展示标准的单元测试编写方式：

```java
class DictTranslatorTest {

    @Test
    void shouldReturnFirstWhenMultiResultPolicyIsFirstMatch() {
        // 1. 准备 Mock 依赖
        JdbcTemplate jdbcTemplate = mock(JdbcTemplate.class);
        when(jdbcTemplate.queryForList(anyString(), eq(Object.class), eq("gender"), eq("男")))
                .thenReturn(List.of("M", "MALE"));

        // 2. 创建被测对象
        DictTranslator translator = new DictTranslator(jdbcTemplate);

        // 3. 构建测试上下文
        TranslateContext context = context("gender", "男");
        context.setMultiResultPolicy(TranslationPolicy.MultiResultPolicy.FIRST_MATCH);

        // 4. 执行并断言
        Object result = translator.translate(context);
        assertEquals("M", result);
    }

    @Test
    void shouldReturnCandidatesWhenMultiResultPolicyIsUserSelect() {
        JdbcTemplate jdbcTemplate = mock(JdbcTemplate.class);
        when(jdbcTemplate.queryForList(anyString(), eq(Object.class), eq("gender"), eq("男")))
                .thenReturn(List.of("M", "MALE"));
        DictTranslator translator = new DictTranslator(jdbcTemplate);

        TranslateContext context = context("gender", "男");
        context.setMultiResultPolicy(TranslationPolicy.MultiResultPolicy.USER_SELECT);

        Object result = translator.translate(context);
        assertEquals(List.of("M", "MALE"), result);
    }

    @Test
    void shouldReturnNullWhenNotFoundPolicyIsSkip() {
        JdbcTemplate jdbcTemplate = mock(JdbcTemplate.class);
        when(jdbcTemplate.queryForList(anyString(), eq(Object.class), eq("gender"), eq("未知")))
                .thenReturn(List.of());
        DictTranslator translator = new DictTranslator(jdbcTemplate);

        TranslateContext context = context("gender", "未知");
        context.setNotFoundPolicy(TranslationPolicy.NotFoundPolicy.SKIP);

        Object result = translator.translate(context);
        assertNull(result);
    }

    @Test
    void shouldThrowWhenMultiResultPolicyIsFail() {
        JdbcTemplate jdbcTemplate = mock(JdbcTemplate.class);
        when(jdbcTemplate.queryForList(anyString(), eq(Object.class), eq("gender"), eq("男")))
                .thenReturn(List.of("M", "MALE"));
        DictTranslator translator = new DictTranslator(jdbcTemplate);

        TranslateContext context = context("gender", "男");
        context.setMultiResultPolicy(TranslationPolicy.MultiResultPolicy.FAIL);

        assertThrows(IllegalArgumentException.class, () -> translator.translate(context));
    }

    // 辅助方法：构建 TranslateContext
    private TranslateContext context(String target, Object sourceValue) {
        TranslateContext context = new TranslateContext();
        context.setTarget(target);
        context.setSourceValue(sourceValue);
        return context;
    }
}
```

**这个测试的关键设计模式**：

1. **Mock 外部依赖**：`JdbcTemplate` 是外部依赖，使用 `Mockito.mock()` 隔离数据库。
2. **覆盖策略分支**：每个 `@Test` 对应一种策略组合（`FIRST_MATCH` / `USER_SELECT` / `SKIP` / `FAIL`），确保各分支都被覆盖。
3. **辅助方法提取**：`context()` 方法减少了测试代码的重复。
4. **断言精确**：使用 `assertEquals` / `assertNull` / `assertThrows` 分别验证正常返回、空值返回和异常抛出。

### 8.3 测试命名规范

采用 `should...When...` 模式，确保测试名称即文档：

```
should<期望行为>When<前置条件>
```

示例：
- `shouldReturnFirstWhenMultiResultPolicyIsFirstMatch`
- `shouldReturnNullWhenNotFoundPolicyIsSkip`
- `shouldThrowWhenMultiResultPolicyIsFail`
- `shouldRejectWhenOverLimit`
- `shouldTrimUnderBudget`

**避免**：
- ❌ `test1`、`testTranslator`、`testMethod`
- ❌ 使用中文方法名（某些 CI 环境可能有兼容问题）

## 11. 常见问题

### 11.1 测试不稳定

**现象**：测试在本地通过，CI 中偶发失败。

**常见原因与解决方案**：

| 原因 | 解决方案 |
|------|---------|
| 依赖外部服务（LLM API、远程数据库） | 使用 Mockito Mock 或 H2 内嵌数据库 |
| 测试间共享可变状态 | 每个测试方法使用 `@BeforeEach` 重置状态 |
| 并发测试互相影响 | 使用 `@Isolated` 注解或独立端口 |
| 时间相关断言不稳定 | 使用 `Awaitility` 做异步等待，避免 `Thread.sleep` |
| 文件路径问题（Windows vs Linux） | 使用 `Path.of()` 而非硬编码分隔符 |

### 11.2 线上问题定位

**定位流程**：

```
1. 用户反馈问题，提供 traceId（或从前端 SSE 事件中提取）
       │
2. 在审计日志中搜索该 traceId
   $ grep "trace-xxx" logs/agent-audit.log | jq .
       │
3. 查看事件序列：是否有 sql_execution 被拒绝？工具调用是否失败？
       │
4. 若涉及性能问题，查看 Actuator 指标
   $ curl localhost:8080/actuator/metrics/agent.tool.duration?tag=tool.name:xxx
       │
5. 若涉及 Token 超限，查看 Token 消耗指标
   $ curl localhost:8080/actuator/metrics/agent.token.consumed
       │
6. 结合 Grafana 面板查看时间区间内的异常趋势
```

**常用排查命令**：

```bash
# 查看最近 100 条审计日志
tail -100 logs/agent-audit.log | jq .

# 查看所有被拒绝的 SQL
grep '"allowed":false' logs/agent-audit.log | jq .

# 统计各工具的调用失败次数
grep '"success":false' logs/agent-audit.log | jq -r '.toolName' | sort | uniq -c | sort -rn

# 查看某个会话的完整事件流
grep "sess-xyz-789" logs/agent-audit.log | jq -s 'sort_by(.timestamp)'
```

### 11.3 指标不显示

**现象**：访问 `/actuator/metrics/agent.tool.calls` 返回 404 或空值。

**排查清单**：

| 检查项 | 说明 |
|-------|------|
| 1. 是否引入了 `spring-boot-starter-actuator` 依赖 | `AgentMetrics` 的自动装配依赖 `MeterRegistry`，没有 Actuator 则不会激活 |
| 2. `management.endpoints.web.exposure.include` 是否包含 `metrics` | 默认只暴露 `health`，需手动添加 |
| 3. 指标是否已被触发过 | Counter/Timer 在首次 `increment()` / `record()` 后才会注册，未调用则不可见 |
| 4. 是否访问了正确的指标名称 | Prometheus 格式下 `.` 会被替换为 `_`，如 `agent_tool_calls_total` |
| 5. 是否使用了正确的 tag 过滤 | 查询时 tag 格式为 `?tag=key:value`，注意冒号分隔 |

**快速验证**：

```bash
# 列出所有已注册的指标名称
curl http://localhost:8080/actuator/metrics | jq '.names[] | select(startswith("agent."))'

# 如果为空，说明 AgentMetrics Bean 未注册，检查自动装配条件
```

### 11.4 审计日志未生成

**排查清单**：

| 检查项 | 说明 |
|-------|------|
| 1. `logback-spring.xml` 是否配置了 `agent-audit` Logger | 缺少配置时日志会走 root logger |
| 2. 是否在业务流程中显式调用了 `AgentAuditLogger` | 该组件不会自动接入，需手动调用 |
| 3. 日志文件路径是否有写入权限 | 检查 `logs/` 目录是否存在且可写 |
| 4. `ObjectMapper` Bean 是否正常 | 序列化失败会输出 WARN 级别日志到控制台 |
