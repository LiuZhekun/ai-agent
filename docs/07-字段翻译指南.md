# 字段翻译指南

## 1. 概述

在 AI Agent 与业务系统交互的场景中，存在一个核心矛盾：**用户用自然语言说话，而系统用编码和 ID 存储数据**。

例如：

| 用户说的话 | 系统需要的值 | 数据来源 |
|---|---|---|
| "性别是男" | `"M"` | `sys_dict` 表中 `type='gender'` 的字典项 |
| "部门是技术部" | `1` (Long) | `sys_department` 表中 `name='技术部'` 的记录 ID |
| "状态为启用" | `1` (Integer) | `sys_dict` 表中 `type='user_status'` 的字典项 |

如果不做翻译，LLM 要么把 `"男"` 直接写入数据库（违反字段约束），要么需要在 system prompt 中硬编码所有字典映射（不可维护）。

**字段翻译模块** (`ai-agent-translator`) 解决的就是这个问题——在 AI Agent 调用业务工具之前，自动将自然语言中的"显示值"映射为数据库所需的"存储值"，做到：

- **对 LLM 透明**：LLM 只需传人类可读的名称，无需理解底层编码体系
- **对工具透明**：工具层收到的参数已经是可直接持久化的值
- **可扩展**：通过 SPI 机制支持自定义翻译类型

## 2. 核心概念

翻译模块由三个核心组件构成，它们之间的关系如下：

```
┌─────────────────────────────────────────────────────┐
│                 TranslatorRegistry                   │
│         (翻译器注册中心，Spring 自动发现)              │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ DictTranslator│  │EntityRefTrans│  │ 自定义翻译器│ │
│  │  type="DICT"  │  │type="ENTITY_ │  │ type="XXX" │ │
│  │               │  │     REF"     │  │            │ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
└──────────────────────┬──────────────────────────────┘
                       │
              translate(TranslateContext)
                       │
              ┌────────▼────────┐
              │ TranslateContext │
              │  ─ sourceValue  │
              │  ─ target       │
              │  ─ lookupField  │
              │  ─ resultField  │
              │  ─ policies     │
              └─────────────────┘
```

### 2.1 FieldTranslator（翻译器 SPI）

`FieldTranslator` 是翻译模块的核心接口，采用策略模式设计。每种翻译类型（字典、实体引用等）对应一个实现类：

```java
public interface FieldTranslator {
    /** 返回翻译器的类型标识，如 "DICT"、"ENTITY_REF" */
    String getType();

    /** 根据上下文执行翻译 */
    Object translate(TranslateContext ctx);
}
```

扩展方式非常简单——实现该接口并标注 `@Component`，即可被框架自动发现和注册。

### 2.2 TranslatorRegistry（翻译器注册中心）

`TranslatorRegistry` 是翻译器的路由中心。启动时，Spring 将所有 `FieldTranslator` 实例注入，注册中心按 `getType()` 返回值建立映射表：

```java
@Component
public class TranslatorRegistry {
    private final Map<String, FieldTranslator> translators;

    public TranslatorRegistry(List<FieldTranslator> translators) {
        this.translators = translators.stream()
            .collect(Collectors.toMap(FieldTranslator::getType, Function.identity()));
    }

    public FieldTranslator get(String type) {
        FieldTranslator translator = translators.get(type);
        if (translator == null) {
            throw new IllegalArgumentException("Translator not found: " + type);
        }
        return translator;
    }
}
```

调用方只需提供类型字符串（如 `"DICT"`），即可获取对应的翻译器实例，无需关心具体实现类。

### 2.3 TranslateContext（翻译上下文）

`TranslateContext` 承载单次翻译调用所需的全部参数：

| 字段 | 类型 | 说明 | 默认值 |
|---|---|---|---|
| `sourceValue` | `Object` | 待翻译的原始值，如用户输入的 `"男"` | — |
| `target` | `String` | 翻译目标标识：字典类型编码或实体表名 | — |
| `lookupField` | `String` | 在目标数据源中用于匹配的列名 | `"name"` |
| `resultField` | `String` | 翻译结果所在的列名 | `"id"` |
| `multiResultPolicy` | `MultiResultPolicy` | 匹配到多条记录时的冲突策略 | `FIRST_MATCH` |
| `notFoundPolicy` | `NotFoundPolicy` | 未匹配到记录时的处理策略 | `FAIL` |
| `metadata` | `Map<String, Object>` | 可扩展的附加信息，供自定义翻译器使用 | — |

## 3. 双路径机制

翻译模块提供两条完全独立的使用路径，适用于不同的集成场景：

### 路径 A：注解驱动 + 装饰器自动翻译

```
用户输入 → LLM 生成工具调用 → 装饰器拦截 → 自动翻译字段 → 工具执行
```

**工作原理**：`TranslateToolCallbackDecorator` 实现了 `ToolCallbackInterceptor` 接口，在工具调用前（`beforeCall`）自动扫描工具输入 JSON，按规则完成翻译。

**翻译触发方式**：

1. **显式规则**——工具输入 JSON 中包含 `_translate` 配置块，显式指定每个字段的翻译规则：
   ```json
   {
     "genderName": "男",
     "deptName": "技术部",
     "_translate": {
       "genderName": { "type": "DICT", "target": "gender" },
       "deptName": { "type": "ENTITY_REF", "target": "sys_department" }
     }
   }
   ```

2. **启发式规则**——根据字段命名约定自动识别：
   - 以 `DictName` 结尾的字段（如 `statusDictName`）自动触发字典翻译，前提是输入中存在对应的 `DictType` 字段（如 `statusDictType`）
   - 已知的实体引用字段名（如 `deptName` → `sys_department`，`userName` → `sys_user`）自动触发实体引用翻译

**适用场景**：
- DTO 字段已通过 `@TranslateField` 注解声明了翻译规则
- 希望翻译过程完全自动，不依赖 LLM 的判断
- 适合字段翻译规则固定且可预测的业务

**特点**：
- 翻译失败不会阻断工具执行（鲁棒性优先）
- 翻译结果通过 `ThreadLocal` 和工具级快照保存，可用于调试

### 路径 B：LLM 自主调用翻译工具

```
用户输入 → LLM 识别到需要翻译 → LLM 主动调用翻译工具 → 获得结果 → LLM 使用翻译后的值调用业务工具
```

**工作原理**：`TranslateTools` 类将翻译能力注册为 LLM 可调用的工具，LLM 在对话过程中自主决定是否需要翻译。

`TranslateTools` 提供两个工具方法：

| 工具方法 | 功能 | 参数 |
|---|---|---|
| `findDictCode` | 字典名称 → 字典编码 | `dictType`（字典类型）, `dictName`（字典名称） |
| `findEntityId` | 实体名称 → 实体主键 ID | `tableName`（目标表名）, `name`（实体名称） |

**适用场景**：
- 翻译需求无法通过注解预定义，需要 LLM 根据上下文动态判断
- 希望 LLM 具备更灵活的翻译决策能力
- 适合开放式对话中翻译规则不确定的场景

### 路径选择建议

| 维度 | 路径 A（装饰器） | 路径 B（LLM 工具） |
|---|---|---|
| 触发方式 | 自动拦截，对 LLM 透明 | LLM 主动调用 |
| 翻译规则 | 预定义（注解/命名约定） | 动态判断 |
| 可控性 | 高，规则固定可预测 | 中，依赖 LLM 判断 |
| Token 消耗 | 无额外消耗 | 每次翻译消耗一次工具调用的 Token |
| 失败处理 | 静默降级 | LLM 可感知并做补救 |

> **推荐做法**：对于已知的、可枚举的翻译字段，优先使用路径 A；对于复杂或动态的翻译需求，两条路径可以共存。

## 4. @TranslateField 注解详解

`@TranslateField` 是声明 DTO 字段自动翻译规则的核心注解，标注在 DTO 字段上。

### 4.1 注解定义

```java
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface TranslateField {
    String type();
    String source();
    String target();
    String lookupField() default "name";
    String resultField() default "id";
    TranslationPolicy.MultiResultPolicy multiResultPolicy()
        default TranslationPolicy.MultiResultPolicy.FIRST_MATCH;
    TranslationPolicy.NotFoundPolicy notFoundPolicy()
        default TranslationPolicy.NotFoundPolicy.FAIL;
}
```

### 4.2 属性说明

| 属性 | 必填 | 说明 | 示例值 |
|---|---|---|---|
| `type` | 是 | 翻译器类型标识，对应 `FieldTranslator.getType()` 的返回值 | `"DICT"`, `"ENTITY_REF"` |
| `source` | 是 | DTO 中提供原始值的字段名（即 JSON 中的 key），通常是人类可读的名称字段 | `"genderName"`, `"departmentName"` |
| `target` | 是 | 翻译目标标识。DICT 类型对应字典 `type` 值；ENTITY_REF 类型对应目标表名 | `"gender"`, `"sys_department"` |
| `lookupField` | 否 | 在目标数据源中用于匹配的列名 | 默认 `"name"` |
| `resultField` | 否 | 翻译结果所在的列名 | DICT 默认 `"code"`，ENTITY_REF 默认 `"id"` |
| `multiResultPolicy` | 否 | 匹配到多条记录时的策略 | 默认 `FIRST_MATCH` |
| `notFoundPolicy` | 否 | 未匹配到记录时的策略 | 默认 `FAIL` |

### 4.3 完整 DTO 示例

以下是 `UserDTO` 的完整示例，展示了两种翻译类型的用法：

```java
public class UserDTO {
    private String name;
    private String phone;
    private String email;

    /**
     * 性别编码。
     * AI 传入 genderName（如"男"），框架自动从 sys_dict 中查找
     * type='gender' & name='男' 对应的 code='M'，赋值到本字段。
     */
    @TranslateField(
        type = "DICT",           // 使用字典翻译器
        source = "genderName",   // 从 JSON 中的 genderName 字段取值
        target = "gender"        // 查询 sys_dict 表中 type='gender' 的记录
    )
    private String gender;

    /**
     * 部门 ID（外键）。
     * AI 传入 departmentName（如"技术部"），框架自动从 sys_department 表中
     * 查找 name='技术部' 对应的 id=1，赋值到本字段。
     */
    @TranslateField(
        type = "ENTITY_REF",         // 使用实体引用翻译器
        source = "departmentName",   // 从 JSON 中的 departmentName 字段取值
        target = "sys_department"    // 查询 sys_department 表
    )
    private Long deptId;

    private Integer status;

    // getter / setter 省略
}
```

**翻译流程图解**：

```
LLM 生成的工具调用参数:
{
  "name": "张三",
  "genderName": "男",            ← source 字段（人类可读）
  "departmentName": "技术部"      ← source 字段（人类可读）
}
            │
            ▼  @TranslateField 注解驱动
┌───────────────────────────┐
│  genderName="男"           │
│  → DICT: sys_dict          │
│    WHERE type='gender'     │
│    AND name='男'           │
│  → code='M'               │
│  → gender = "M"            │
├───────────────────────────┤
│  departmentName="技术部"    │
│  → ENTITY_REF:             │
│    sys_department           │
│    WHERE name='技术部'      │
│  → id=1                    │
│  → deptId = 1L             │
└───────────────────────────┘
            │
            ▼
工具接收到的实际参数:
{
  "name": "张三",
  "gender": "M",              ← 已翻译为存储值
  "deptId": 1                  ← 已翻译为存储值
}
```

## 5. 内置翻译器

### 5.1 DictTranslator（字典表翻译器）

**类型标识**：`"DICT"`

**功能**：查询 `sys_dict` 表，将字典名称翻译为对应的字典编码。

**查询逻辑**：

```sql
SELECT <resultField> FROM sys_dict WHERE type = ? AND <lookupField> = ?
-- 默认: SELECT code FROM sys_dict WHERE type = ? AND name = ?
```

**参数映射**：

| TranslateContext 字段 | 对应 SQL | 默认值 |
|---|---|---|
| `target` | `WHERE type = ?` | —（必填） |
| `sourceValue` | `WHERE name = ?` | —（必填） |
| `lookupField` | 匹配列名 | `"name"` |
| `resultField` | 结果列名 | `"code"` |

**典型场景**：

```
用户说"状态为启用"
  → sourceValue = "启用"
  → target = "user_status"
  → SQL: SELECT code FROM sys_dict WHERE type='user_status' AND name='启用'
  → 返回: 1
```

**前置条件**：数据库中需要存在 `sys_dict` 表，结构至少包含以下列：

```sql
CREATE TABLE sys_dict (
    id       BIGINT PRIMARY KEY AUTO_INCREMENT,
    type     VARCHAR(64)  NOT NULL COMMENT '字典类型',
    name     VARCHAR(128) NOT NULL COMMENT '字典名称（显示值）',
    code     VARCHAR(128) NOT NULL COMMENT '字典编码（存储值）',
    sort     INT DEFAULT 0 COMMENT '排序号'
);
```

### 5.2 EntityRefTranslator（实体引用翻译器）

**类型标识**：`"ENTITY_REF"`

**功能**：将实体名称翻译为对应的主键 ID，适用于外键引用场景。

**查询逻辑**：

```sql
SELECT <resultField> FROM <target_table> WHERE <lookupField> = ?
-- 默认: SELECT id FROM sys_department WHERE name = ?
```

**参数映射**：

| TranslateContext 字段 | 对应 SQL | 默认值 |
|---|---|---|
| `target` | 目标表名 | —（必填，需在白名单内） |
| `sourceValue` | `WHERE name = ?` | —（必填） |
| `lookupField` | 匹配列名 | `"name"` |
| `resultField` | 结果列名 | `"id"` |

**典型场景**：

```
用户说"部门为研发部"
  → sourceValue = "研发部"
  → target = "sys_department"
  → SQL: SELECT id FROM sys_department WHERE name = '研发部'
  → 返回: 2
```

**表白名单机制**：`EntityRefTranslator` 内置了一个静态白名单，只有白名单中的表才允许查询：

```java
private static final Map<String, String> TABLE_WHITELIST = Map.of(
    "sys_department", "sys_department",
    "sys_user", "sys_user"
);
```

需要支持新的实体表时，必须在白名单中显式注册（详见[第 9 节：安全考虑](#9-安全考虑)）。

## 6. 冲突与失败策略

翻译过程中可能遇到两种异常情况：**匹配到多条记录**和**未匹配到任何记录**。`TranslationPolicy` 提供了完整的策略枚举来处理这两种场景。

### 6.1 多结果策略（MultiResultPolicy）

当翻译查询返回多条匹配记录时生效：

| 策略 | 行为 | 适用场景 |
|---|---|---|
| `FIRST_MATCH` | 取第一条匹配结果 | 数据质量可控，重名概率低，追求简单高效 |
| `USER_SELECT` | 将所有候选项返回给调用方（返回 `List`） | 数据可能有重名（如同名部门），需要用户在前端选择 |
| `FAIL` | 直接抛出 `IllegalArgumentException` 异常 | 严格模式，数据不应出现重复，遇到即视为数据异常 |

**注解配置示例**：

```java
@TranslateField(
    type = "ENTITY_REF",
    source = "departmentName",
    target = "sys_department",
    multiResultPolicy = TranslationPolicy.MultiResultPolicy.USER_SELECT
)
private Long deptId;
```

### 6.2 未命中策略（NotFoundPolicy）

当翻译查询未返回任何匹配记录时生效：

| 策略 | 行为 | 适用场景 |
|---|---|---|
| `FAIL` | 抛出 `IllegalArgumentException` 异常 | 严格模式，翻译目标必须存在，否则整个操作应中断 |
| `SKIP` | 返回 `null`，跳过该字段的翻译 | 字段非必填，翻译失败不影响整体流程 |
| `USE_ORIGINAL` | 保留原始输入值不做转换 | 允许用户直接输入编码值（兼容模式） |

**注解配置示例**：

```java
@TranslateField(
    type = "DICT",
    source = "statusName",
    target = "user_status",
    notFoundPolicy = TranslationPolicy.NotFoundPolicy.USE_ORIGINAL
)
private Integer status;
```

### 6.3 策略组合建议

| 业务场景 | multiResultPolicy | notFoundPolicy | 说明 |
|---|---|---|---|
| 数据严格（如状态字典） | `FIRST_MATCH` | `FAIL` | 字典数据质量高，不会有重名，未找到即数据有误 |
| 外键引用（如部门） | `USER_SELECT` | `FAIL` | 可能同名部门，让用户选择；但部门必须存在 |
| 可选字段（如标签） | `FIRST_MATCH` | `SKIP` | 翻译失败不影响主流程 |
| 兼容模式（接受原始值） | `FIRST_MATCH` | `USE_ORIGINAL` | 用户可能直接输入编码值 |

## 7. 自定义翻译器开发

当内置的 `DictTranslator` 和 `EntityRefTranslator` 无法满足需求时（如需要调用远程 API、查询 Redis 缓存、或使用自定义编码规则），可以开发自定义翻译器。

### 7.1 实现步骤

1. 创建类，实现 `FieldTranslator` 接口
2. 标注 `@Component` 注解，让 Spring 自动发现
3. 在 `getType()` 中返回全局唯一的类型标识
4. 在 `translate()` 中实现翻译逻辑

### 7.2 完整代码示例

以下示例实现了一个通过 Redis 缓存进行翻译的翻译器：

```java
package com.example.translator;

import io.github.aiagent.translator.FieldTranslator;
import io.github.aiagent.translator.TranslateContext;
import io.github.aiagent.translator.strategy.TranslationPolicy;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Component;

@Component
public class RedisCacheTranslator implements FieldTranslator {

    private final StringRedisTemplate redisTemplate;

    public RedisCacheTranslator(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    @Override
    public String getType() {
        return "REDIS_CACHE";
    }

    @Override
    public Object translate(TranslateContext ctx) {
        // target 作为 Redis key 的前缀，sourceValue 作为具体的 key
        String redisKey = ctx.getTarget() + ":" + ctx.getSourceValue();
        String result = redisTemplate.opsForValue().get(redisKey);

        if (result == null) {
            return handleNotFound(ctx);
        }
        return result;
    }

    private Object handleNotFound(TranslateContext ctx) {
        TranslationPolicy.NotFoundPolicy policy = ctx.getNotFoundPolicy();
        if (policy == TranslationPolicy.NotFoundPolicy.USE_ORIGINAL) {
            return ctx.getSourceValue();
        }
        if (policy == TranslationPolicy.NotFoundPolicy.SKIP) {
            return null;
        }
        throw new IllegalArgumentException(
            "Redis cache translation not found: " + ctx.getTarget() + ":" + ctx.getSourceValue()
        );
    }
}
```

### 7.3 在 DTO 中使用自定义翻译器

```java
@TranslateField(
    type = "REDIS_CACHE",         // 对应 getType() 的返回值
    source = "regionName",        // JSON 中的源字段名
    target = "region_mapping",    // Redis key 前缀
    notFoundPolicy = TranslationPolicy.NotFoundPolicy.SKIP
)
private String regionCode;
```

### 7.4 利用 metadata 扩展

`TranslateContext` 中的 `metadata` 字段为自定义翻译器预留了扩展空间。当注解提供的标准属性不足以表达翻译规则时，可以通过路径 B（LLM 工具模式）手动构建 `TranslateContext` 并注入自定义参数：

```java
TranslateContext ctx = new TranslateContext();
ctx.setSourceValue("华东区");
ctx.setTarget("region_mapping");
ctx.setMetadata(Map.of(
    "fuzzyMatch", true,        // 自定义：是否模糊匹配
    "maxResults", 5            // 自定义：最大返回条数
));
translatorRegistry.get("REDIS_CACHE").translate(ctx);
```

## 8. 配置参考

### 8.1 Maven 依赖

在业务模块的 `pom.xml` 中引入 translator 模块：

```xml
<dependency>
    <groupId>io.github.aiagent</groupId>
    <artifactId>ai-agent-translator</artifactId>
    <version>${project.version}</version>
</dependency>
```

`ai-agent-translator` 自身依赖了 `ai-agent-core` 和 `spring-boot-starter-jdbc`，会自动传递引入。

### 8.2 数据源配置

翻译器通过 `JdbcTemplate` 查询数据库，因此需要在 `application.yml` 中配置数据源：

```yaml
spring:
  datasource:
    url: jdbc:mysql://127.0.0.1:3306/ai_agent?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
    username: root
    password: root
    driver-class-name: com.mysql.cj.jdbc.Driver
```

### 8.3 工具调用配置

翻译装饰器与工具系统集成，相关超时和重试配置位于：

```yaml
ai:
  agent:
    tool:
      timeout-seconds: 10    # 单次工具调用超时（包含翻译耗时）
      max-retries: 1         # 工具调用失败后重试次数
```

### 8.4 翻译器自动注册

翻译器的注册完全依赖 Spring 自动发现机制，**无需额外的 YAML 配置**。只要翻译器类：

1. 实现了 `FieldTranslator` 接口
2. 标注了 `@Component`（或其他 Spring Bean 注解）
3. 位于 Spring 组件扫描路径内

就会在应用启动时自动注册到 `TranslatorRegistry` 中。

## 9. 安全考虑

翻译模块直接与数据库交互，存在潜在的安全风险。框架在设计上采取了多重防护措施。

### 9.1 表白名单机制

`EntityRefTranslator` 维护了一个静态白名单 `TABLE_WHITELIST`，只有白名单中的表才允许查询。当请求非白名单表时，会立即抛出异常：

```java
String table = TABLE_WHITELIST.get(ctx.getTarget());
if (table == null) {
    throw new IllegalArgumentException("Target table not allowed: " + ctx.getTarget());
}
```

**扩展白名单**：当需要支持新的实体表时，需修改 `EntityRefTranslator` 中的 `TABLE_WHITELIST`：

```java
private static final Map<String, String> TABLE_WHITELIST = Map.of(
    "sys_department", "sys_department",
    "sys_user", "sys_user",
    "sys_role", "sys_role"          // 新增：允许角色表参与翻译
);
```

> **重要**：绝不要将白名单设计为"允许所有表"。白名单机制是防止 LLM 被注入恶意 prompt 后访问敏感表（如 `sys_config`、`sys_password` 等）的关键屏障。

### 9.2 SQL 注入防护

翻译器在两个层面防止 SQL 注入：

**1. 参数化查询**：所有用户输入值（`sourceValue`）通过 `JdbcTemplate` 的参数化查询传入，不会直接拼接到 SQL 中：

```java
// 安全：sourceValue 作为参数绑定
jdbcTemplate.queryForList(sql, Object.class, ctx.getTarget(), String.valueOf(ctx.getSourceValue()));
```

**2. 标识符校验**：`lookupField` 和 `resultField` 作为列名需要拼接到 SQL 中，因此采用正则白名单校验，仅允许合法的 SQL 标识符：

```java
private String identifierOrDefault(String input, String defaultValue) {
    if (input == null || input.isBlank()) {
        return defaultValue;
    }
    // 仅允许字母、数字、下划线，且必须以字母或下划线开头
    if (!input.matches("[a-zA-Z_][a-zA-Z0-9_]*")) {
        throw new IllegalArgumentException("Invalid field identifier: " + input);
    }
    return input;
}
```

### 9.3 安全最佳实践

| 实践 | 说明 |
|---|---|
| 最小化白名单 | 只将确实需要参与翻译的表加入白名单 |
| 避免暴露敏感列 | 不要将 `password`、`secret` 等列设为 `resultField` |
| 生产环境数据库账号权限 | 为翻译模块使用只读数据库账号，限制 `SELECT` 权限范围 |
| 定期审计 | 检查白名单中的表是否仍有翻译需求，及时清理不再使用的表 |

### 步骤 3：定义 DTO 并标注注解

```java
import io.github.aiagent.translator.annotation.TranslateField;
import io.github.aiagent.translator.strategy.TranslationPolicy;

public class UserDTO {
    private String name;
    private String phone;
    private String email;

    @TranslateField(
        type = "DICT",
        source = "genderName",
        target = "gender"
    )
    private String gender;

    @TranslateField(
        type = "ENTITY_REF",
        source = "departmentName",
        target = "sys_department",
        multiResultPolicy = TranslationPolicy.MultiResultPolicy.USER_SELECT,
        notFoundPolicy = TranslationPolicy.NotFoundPolicy.FAIL
    )
    private Long deptId;

    @TranslateField(
        type = "DICT",
        source = "statusName",
        target = "user_status",
        notFoundPolicy = TranslationPolicy.NotFoundPolicy.USE_ORIGINAL
    )
    private Integer status;

    // getter / setter 省略
}
```

框架自带的 `DictTranslator` 和 `EntityRefTranslator` 都标注了 `@Component`，只要 `ai-agent-translator` 在 classpath 中且包路径在组件扫描范围内，它们会自动注册。

## 11. 常见问题

### Q1: 翻译未生效，工具收到的仍是原始值

**排查步骤**：

1. **确认依赖引入**：检查业务模块的 `pom.xml` 是否引入了 `ai-agent-translator`
2. **确认组件扫描**：翻译器类是否在 Spring 组件扫描路径内。如果翻译器在独立模块中，确认主应用的 `@ComponentScan` 或 `@SpringBootApplication` 覆盖了翻译器所在的包路径
3. **确认注解标注**：DTO 字段上的 `@TranslateField` 中 `source` 值是否与 LLM 传入的 JSON key 一致
4. **查看日志**：搜索 `Tool pre-translation` 关键字，查看装饰器是否成功执行了翻译
5. **检查数据**：确认 `sys_dict` 表中确实存在匹配的字典数据

### Q2: 翻译返回了多个结果，如何处理？

当查询返回多条匹配记录时，行为取决于 `multiResultPolicy` 的设置：

- **`FIRST_MATCH`（默认）**：静默取第一条，适合数据质量较高的场典
- **`USER_SELECT`**：返回候选列表（`List`），由上层逻辑（如前端弹窗）让用户选择。使用此策略时，工具层需要处理返回值为 `List` 的情况
- **`FAIL`**：直接抛异常中断操作，适合严格不允许歧义的场景

**建议**：对于外键引用类字段（如部门、角色），推荐使用 `USER_SELECT`，避免在有同名记录时误写入。

### Q3: 字典中找不到对应的值怎么办？

未命中的行为取决于 `notFoundPolicy` 的设置：

- **`FAIL`（默认）**：抛出 `IllegalArgumentException`，中断操作。LLM 会收到错误信息并告知用户
- **`SKIP`**：返回 `null`，字段保持未赋值状态。适用于非必填字段
- **`USE_ORIGINAL`**：保留用户原始输入值。适用于兼容模式——用户可能直接输入了编码值而不是名称

### Q4: 如何支持新的实体表翻译？

需要同时做两件事：

1. **白名单注册**：在 `EntityRefTranslator.TABLE_WHITELIST` 中添加新表
2. **DTO 注解**：在相应的 DTO 字段上添加 `@TranslateField` 注解

### Q5: 自定义翻译器未被识别

确认以下几点：

1. 类上标注了 `@Component`（或 `@Service` 等）
2. `getType()` 返回的类型标识全局唯一，未与已有翻译器冲突
3. 类所在的包在 Spring 组件扫描范围内

### Q6: 翻译过程中发生异常会阻断业务吗？

取决于使用的路径：

- **路径 A（装饰器）**：不会。`TranslateToolCallbackDecorator` 的 `beforeCall` 方法用 try-catch 包裹了整个翻译流程，异常只会被记录为警告日志（`Tool pre-translation skipped`），不会阻断工具执行
- **路径 B（LLM 工具）**：会。LLM 调用翻译工具时，如果翻译失败会收到错误信息。但 LLM 通常会根据错误信息进行补救（如提示用户确认）

### Q7: 如何调试翻译结果？

`TranslateToolCallbackDecorator` 提供了 `getLastToolTranslations(String toolName)` 方法，可获取某个工具最近一次调用的翻译快照：

```java
@Autowired
private TranslateToolCallbackDecorator decorator;

// 获取最近一次 createUser 工具调用的翻译结果
Map<String, Object> translations = decorator.getLastToolTranslations("createUser");
// {genderName=M, departmentName=1}
```

此方法在单元测试和问题排查中非常有用。
