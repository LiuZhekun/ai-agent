## 1. 概述

在 AI Agent 架构中，**工具（Tool）** 是 Agent 与外部世界交互的唯一桥梁。LLM 本身不能直接读写数据库、调用接口或执行业务逻辑，它只能通过 **Function Calling** 机制向框架发出"我想调用某个工具"的请求，由框架代为执行并将结果返回给 LLM。

一个典型的对话流程如下：

```
用户: "帮我查一下张三的信息"
  ↓
LLM 分析意图，决定调用 queryUser(name="张三")
  ↓
框架执行 queryUser 方法，查询数据库
  ↓
结果经 ToolResultFormatter 格式化后返回给 LLM
  ↓
LLM 根据结果生成用户可读的回复
```

工具开发的质量直接影响 Agent 的能力边界和调用准确率。本文档将从注解定义、注册流程、参数设计、治理能力等维度，详细说明如何开发一个生产级的 Agent 工具。

### 前置条件

- 已引入 `ai-agent-core` 依赖或 `ai-agent-spring-boot-starter`
- 了解 Spring Bean 生命周期与依赖注入
- 了解 Spring AI 的 `@Tool` 注解基本用法

## 2. 核心概念

### 2.1 @AgentTool 与 @Tool 的关系

框架采用**双层注解**设计，两者各司其职：

| 注解 | 级别 | 来源 | 职责 |
|------|------|------|------|
| `@AgentTool` | 类级别 | ai-agent-core | 标记 Bean 为工具组，提供分组名称、描述和风险等级 |
| `@Tool` | 方法级别 | Spring AI | 标记具体方法为可被 LLM 调用的工具函数 |

**`@AgentTool` 不负责工具方法的注册**。它的作用是：

1. **自动注册为 Spring Bean** — 内含 `@Component` 元注解，无需额外标注
2. **元数据分组** — 通过 `name` 和 `description` 为一组工具方法提供统一标识
3. **风险标识** — 通过 `riskLevel` 声明该工具组的操作风险等级

`@Tool` 才是真正让方法成为"可调用工具"的注解。Spring AI 的 `MethodToolCallbackProvider` 会扫描 Bean 上所有 `@Tool` 方法，为每个方法生成一个 `ToolCallback`。

```java
// @AgentTool = 类级别元数据（分组名、描述、风险等级）
@AgentTool(name = "user-tools", description = "用户管理工具")
public class UserTools {

    // @Tool = 方法级别（这才是 LLM 真正能调用的工具）
    @Tool(description = "按姓名/手机号查询用户")
    public List<User> queryUser(String name, String phone) { ... }
}
```

### 2.2 Spring AI ToolCallbackProvider 机制

Spring AI 定义了 `ToolCallbackProvider` 接口，用于向 `ChatClient` 提供可用的工具列表：

```java
public interface ToolCallbackProvider {
    ToolCallback[] getToolCallbacks();
}
```

框架通过 `AgentToolCallbackProvider` 实现了该接口，并标注 `@Primary` 覆盖 Spring AI 的默认实现。这意味着所有注入 `ToolCallbackProvider` 的地方都会得到经过**装饰器增强**的版本，而非原始的 `ToolCallback`。

### 2.3 ToolMetadata — 工具元数据

每个 `@AgentTool` Bean 在扫描时会同步生成 `ToolMetadata` 对象，包含：

- `groupName` — 工具组名称（对应 `@AgentTool.name`）
- `groupDescription` — 工具组描述
- `toolName` — 工具类名
- `riskLevel` — 风险等级（`LOW` / `MEDIUM` / `HIGH`）
- `parameters` — 参数列表（`ParamInfo`：名称、类型、描述、是否必填）
- `beanClass` — Bean 的 Class 对象

元数据服务于多个上层组件：`ClarificationAdvisor` 据此做参数完整性检查；`PlanningAdvisor` 据此做任务分解；管理端点据此展示工具清单。

## 3. 工具注册流程

整个注册过程发生在 `AgentToolCallbackProvider` 的构造函数中，**应用启动时一次性完成**，运行时不会再动态扫描。

```
应用启动
  │
  ▼
AgentToolCallbackProvider 构造
  │
  ├─ 1. 扫描：从 ApplicationContext 获取所有 @AgentTool Bean
  │
  ├─ 2. 提取：对每个 Bean 用 MethodToolCallbackProvider 提取 @Tool 方法
  │     生成原始 ToolCallback[]
  │
  ├─ 3. 装饰：将每个 ToolCallback 包装为 ToolCallbackDecorator
  │     注入限流 Semaphore、超时、重试、拦截器链、结果格式化器
  │
  ├─ 4. 元数据：同步构建 ToolMetadata 列表
  │     供 ClarificationAdvisor / PlanningAdvisor / 管理端点使用
  │
  ▼
注册完成，getToolCallbacks() 返回装饰后的数组
```

### 关键源码解读

```java
// AgentToolCallbackProvider 构造函数核心逻辑
Map<String, Object> beans = applicationContext.getBeansWithAnnotation(AgentTool.class);
Semaphore semaphore = new Semaphore(5); // 全局并发上限

for (Object bean : beans.values()) {
    AgentTool annotation = AnnotatedElementUtils.findMergedAnnotation(
        bean.getClass(), AgentTool.class);

    // 构建元数据
    ToolMetadata metadata = new ToolMetadata();
    metadata.setGroupName(annotation.name());
    metadata.setRiskLevel(annotation.riskLevel());
    // ...

    // 使用 Spring AI 提取 @Tool 方法
    ToolCallbackProvider methodProvider = MethodToolCallbackProvider.builder()
            .toolObjects(bean)
            .build();

    // 逐个装饰
    for (ToolCallback cb : methodProvider.getToolCallbacks()) {
        callbackList.add(new ToolCallbackDecorator(
                cb, interceptors, metrics, 10, 1, semaphore, formatter));
    }
}
```

注意事项：

- `@AgentTool` 内含 `@Component`，所以标注后的类会自动被 Spring 扫描为 Bean
- 如果类不在组件扫描路径下，工具不会被注册
- 所有 `ToolCallbackInterceptor` Bean 会在构造时自动收集并注入到每个装饰器中

## 4. 开发步骤（详细版）

### 第一步：创建工具类

在业务模块中创建一个 Java 类，标注 `@AgentTool` 注解：

```java
package io.github.aiagent.demo.tools;

import io.github.aiagent.core.tool.annotation.AgentTool;

@AgentTool(
    name = "order-tools",          // 工具组名称，需全局唯一
    description = "订单管理工具",    // 描述会用于元数据展示
    riskLevel = AgentTool.RiskLevel.MEDIUM  // 风险等级
)
public class OrderTools {
    // ...
}
```

`name` 的命名规范：

- 使用小写字母 + 连字符，如 `order-tools`、`user-tools`
- 需与配置白名单中的名称一致
- 建议以 `-tools` 结尾，保持统一风格

### 第二步：注入业务依赖

通过构造函数注入所需的 Service 层依赖：

```java
@AgentTool(name = "order-tools", description = "订单管理工具")
public class OrderTools {

    private final OrderService orderService;

    public OrderTools(OrderService orderService) {
        this.orderService = orderService;
    }
}
```

> 工具类本身是 Spring Bean（`@AgentTool` 包含 `@Component`），因此支持标准的依赖注入。

### 第三步：编写工具方法

在类中添加 `@Tool` 方法。**`description` 是最重要的属性**——它会作为 Function Calling 的描述发送给 LLM，直接影响 AI 调用的准确率。

```java
@Tool(description = "按订单号或客户姓名查询订单，支持模糊匹配")
public List<Order> queryOrder(String orderNo, String customerName) {
    return orderService.query(orderNo, customerName);
}
```

`description` 编写原则：

- 用一句话说清功能，包含关键的参数含义
- 避免过于技术化的描述（LLM 需要理解语义）
- 如果有特殊行为（如"传 null 时查全部"），可在描述中提及

### 第四步：添加到工具白名单

在业务配置中将新工具的 `name` 加入白名单：

```yaml
ai:
  agent:
    demo:
      tool-whitelist:
        - user-tools
        - department-tools
        - dict-tools
        - order-tools        # 新增工具
```

或在配置类 `DemoAgentConfig` 中确认白名单包含该名称。

### 第五步：验证注册

启动应用后，调用管理接口验证工具已注册：

```bash
curl http://localhost:8080/api/agent/tools
```

返回 JSON 中应包含你新增的工具及其方法列表。

## 5. 参数设计最佳实践

### 5.1 DTO vs 松散参数

工具方法的参数有两种风格：

**松散参数** — 适合参数少（1-3 个）、语义简单的查询场景：

```java
@Tool(description = "按姓名/手机号查询用户")
public List<User> queryUser(String name, String phone) {
    return userService.query(name, phone);
}
```

**DTO 参数** — 适合参数多、涉及字段翻译或需要校验的写操作场景：

```java
@Tool(description = "新增用户")
public String addUser(UserDTO dto) {
    User user = convert(dto);
    userService.add(user);
    return "ok";
}
```

选择建议：

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 查询，1-3 个过滤条件 | 松散参数 | 简洁直观，LLM 容易正确填充 |
| 写操作，字段较多 | DTO | 结构清晰，支持 `@TranslateField` 等注解 |
| 字段需要翻译（名称→ID） | DTO | 只有 DTO 才支持 `@TranslateField` |
| 需要参数校验 | DTO | 可在 DTO 上标注 Bean Validation 注解 |

### 5.2 参数命名

参数名会作为 JSON Schema 的字段名传给 LLM，**必须具有语义**：

```java
// 好：LLM 能理解参数含义
public List<User> queryUser(String name, String phone)

// 差：LLM 无法理解 s 和 p 分别代表什么
public List<User> queryUser(String s, String p)
```

### 5.3 可选参数处理

对于可选参数，直接使用可空类型。当 LLM 没有提供某个参数时，Spring AI 会传入 `null`：

```java
@Tool(description = "按姓名/手机号查询用户，参数均为可选，传空时忽略该条件")
public List<User> queryUser(String name, String phone) {
    // name 或 phone 为 null 时忽略对应条件
    return userService.query(name, phone);
}
```

### 5.4 使用 @ToolParam 细化参数描述

Spring AI 提供了 `@ToolParam` 注解，用于为参数添加描述和是否必填标识：

```java
@Tool(description = "按条件查询订单")
public List<Order> queryOrder(
        @ToolParam(description = "订单号，支持模糊匹配", required = false) String orderNo,
        @ToolParam(description = "客户姓名", required = false) String customerName,
        @ToolParam(description = "订单状态：PENDING/PAID/SHIPPED", required = false) String status) {
    return orderService.query(orderNo, customerName, status);
}
```

`@ToolParam` 的信息会被编入 Function Calling 的 JSON Schema，帮助 LLM 更准确地理解每个参数的用途和约束。

## 6. 返回值设计

### 6.1 ToolResultFormatter 处理逻辑

所有工具的返回值都会经过 `ToolResultFormatter` 统一格式化后再传递给 LLM。格式化器根据运行时类型自动选择最佳表示形式：

| 返回类型 | 格式化结果 | 适用场景 |
|----------|-----------|----------|
| `List<Map<String, Object>>` | Markdown 表格 | 结构化查询结果（如用户列表） |
| `Map` / `List`（其他形式） | JSON 文本 | 嵌套结构或非表格数据 |
| `String` / 基本类型 | `String.valueOf()` 纯文本 | 操作结果、简单消息 |
| `null` | `"null"` | 无返回值 |

### 6.2 FormatHint 枚举

格式化器支持四种提示模式：

- `AUTO` — 自动推断（默认使用，框架自动调用）
- `PLAIN_TEXT` — 强制纯文本
- `MARKDOWN_TABLE` — 强制 Markdown 表格
- `JSON` — 强制 JSON 格式

当前框架默认使用 `AUTO` 模式，开发者无需手动干预。

### 6.3 返回值设计建议

**查询操作** — 返回 `List<Entity>` 或 `List<VO>`，框架会自动格式化为表格：

```java
@Tool(description = "按姓名/手机号查询用户")
public List<User> queryUser(String name, String phone) {
    return userService.query(name, phone);
}
```

**写操作** — 返回简短字符串（如 `"ok"`），AI 会据此生成用户可读的回复：

```java
@Tool(description = "新增用户")
public String addUser(UserDTO dto) {
    userService.add(convert(dto));
    return "ok";
}
```

**单对象查询** — 返回 Entity 或 VO 对象，会被序列化为 JSON：

```java
@Tool(description = "按ID查询用户详情")
public User getUserById(Long id) {
    return userService.getById(id);
}
```

**需要避免的做法：**

- 不要返回大段拼接的字符串，让格式化器自动处理
- 不要返回超大列表（考虑分页或限制条数），避免 Token 浪费
- 不要返回敏感信息（如密码哈希），工具层应做脱敏

## 7. 装饰器链

`ToolCallbackDecorator` 是整个工具治理的核心。它以装饰器模式包装原始 `ToolCallback`，在不修改业务代码的前提下叠加五项企业级治理能力。

### 7.1 完整调用链路

```
客户端请求
  │
  ▼
并发闸门（Semaphore.tryAcquire）
  │ 失败 → 抛出 ToolExecutionException("Tool concurrency limit reached")
  ▼
拦截器链 beforeCall()
  │ 按 Spring Bean 顺序依次执行
  ▼
异步超时执行 delegate.call()
  │ 包装为 CompletableFuture，受 timeoutSeconds 限制
  │ 失败 → 重试（最多 maxRetries 次）
  │ 重试耗尽 → 抛出 ToolExecutionException("Tool execution failed after retries")
  ▼
拦截器链 afterCall()
  │ 可对输出做后处理
  ▼
ToolResultFormatter 统一格式化
  │
  ▼
AgentMetrics 指标记录
  │ 记录工具名称、成功/失败、耗时
  ▼
释放闸门（Semaphore.release，在 finally 中）
```

### 7.2 五项治理能力

#### (1) 并发限流

通过共享 `Semaphore` 控制全局工具并发数，防止单工具洪峰击穿下游服务。

- 默认并发上限：**5**
- 超限行为：立即抛出异常，不排队等待
- 配置项：`ai.agent.tool.max-concurrent`

```yaml
ai:
  agent:
    tool:
      max-concurrent: 5   # 最大并发工具调用数
```

#### (2) 超时控制

通过 `CompletableFuture.get(timeout, TimeUnit.SECONDS)` 限制单次调用最大耗时，避免长时间阻塞。

- 默认超时：**10 秒**
- 超时后：触发重试或抛出异常
- 配置项：`ai.agent.tool.timeout-seconds`

```yaml
ai:
  agent:
    tool:
      timeout-seconds: 10  # 单次工具调用超时（秒）
```

#### (3) 自动重试

对瞬态错误（如网络抖动、数据库临时不可用）进行透明重试。

- 默认重试次数：**1**（即最多执行 2 次：首次 + 1 次重试）
- 重试条件：任何异常
- 配置项：`ai.agent.tool.max-retries`

```yaml
ai:
  agent:
    tool:
      max-retries: 1  # 工具调用失败后重试次数
```

#### (4) 拦截器链

调用前后分别执行 `ToolCallbackInterceptor` 链，支持日志、审计、参数改写等扩展。详见[第 8 节](#8-拦截器扩展点)。

#### (5) 结果格式化

通过 `ToolResultFormatter` 将结果转为 LLM 友好文本。详见[第 6 节](#6-返回值设计)。

### 7.3 装饰器参数一览

| 参数 | 类型 | 含义 | 默认值 |
|------|------|------|--------|
| `delegate` | `ToolCallback` | 被装饰的原始回调 | — |
| `interceptors` | `List<ToolCallbackInterceptor>` | 拦截器链 | 自动注入所有 Bean |
| `metrics` | `AgentMetrics` | 指标采集器（可为 null） | 自动注入 |
| `timeoutSeconds` | `int` | 超时秒数 | 10 |
| `maxRetries` | `int` | 最大重试次数 | 1 |
| `concurrencyLimiter` | `Semaphore` | 并发限流器 | permits=5 |
| `formatter` | `ToolResultFormatter` | 结果格式化器 | 自动注入 |

## 8. 拦截器扩展点

`ToolCallbackInterceptor` 是框架提供的 SPI 扩展接口，允许开发者在工具调用前后注入自定义逻辑。

### 8.1 接口定义

```java
public interface ToolCallbackInterceptor {

    /**
     * 工具调用前置拦截。
     * 可用于日志记录、参数校验或访问控制。
     * 若需中断调用，直接抛出运行时异常。
     */
    void beforeCall(String toolName, String toolInput);

    /**
     * 工具调用后置拦截。
     * 可对输出结果进行改写、脱敏或追加信息。
     * 返回值将替换原始输出继续传递给后续拦截器和格式化器。
     */
    String afterCall(String toolName, String toolInput, String toolOutput);
}
```

### 8.2 接入方式

实现接口并注册为 Spring Bean 即可自动生效：

```java
@Component
public class AuditToolInterceptor implements ToolCallbackInterceptor {

    private static final Logger log = LoggerFactory.getLogger(AuditToolInterceptor.class);

    @Override
    public void beforeCall(String toolName, String toolInput) {
        log.info("[工具审计] 调用开始 tool={}, input={}", toolName, toolInput);
    }

    @Override
    public String afterCall(String toolName, String toolInput, String toolOutput) {
        log.info("[工具审计] 调用结束 tool={}, output长度={}", toolName, toolOutput.length());
        return toolOutput;
    }
}
```

`AgentToolCallbackProvider` 会在构造时自动收集所有 `ToolCallbackInterceptor` Bean，并注入到每个 `ToolCallbackDecorator` 中。多个拦截器按 Spring Bean 的默认排序执行（可通过 `@Order` 控制顺序）。

### 8.3 典型使用场景

#### 场景一：调用日志审计

记录每次工具调用的入参、出参和耗时，用于运维排查和安全审计：

```java
@Component
@Order(1)
public class LoggingInterceptor implements ToolCallbackInterceptor {

    @Override
    public void beforeCall(String toolName, String toolInput) {
        MDC.put("toolName", toolName);
        log.info("工具调用开始: {}", toolInput);
    }

    @Override
    public String afterCall(String toolName, String toolInput, String toolOutput) {
        log.info("工具调用结束: output={}", toolOutput);
        MDC.remove("toolName");
        return toolOutput;
    }
}
```

#### 场景二：参数脱敏

在前置拦截中对敏感字段（如手机号、身份证号）加密：

```java
@Component
@Order(2)
public class SensitiveFieldInterceptor implements ToolCallbackInterceptor {

    @Override
    public void beforeCall(String toolName, String toolInput) {
        // 日志中脱敏，实际传给工具的 toolInput 不变
        log.info("工具输入（脱敏）: {}", maskSensitiveFields(toolInput));
    }

    @Override
    public String afterCall(String toolName, String toolInput, String toolOutput) {
        return toolOutput;
    }
}
```

#### 场景三：访问控制

在前置拦截中检查当前用户是否有权调用某工具，不满足时抛出异常中断调用：

```java
@Component
@Order(0)
public class AccessControlInterceptor implements ToolCallbackInterceptor {

    @Override
    public void beforeCall(String toolName, String toolInput) {
        String currentUser = SecurityContextHolder.getContext().getAuthentication().getName();
        if (!hasPermission(currentUser, toolName)) {
            throw new RuntimeException("无权调用工具: " + toolName);
        }
    }

    @Override
    public String afterCall(String toolName, String toolInput, String toolOutput) {
        return toolOutput;
    }
}
```

#### 场景四：结果改写

在后置拦截中对输出做后处理，例如追加额外信息或脱敏：

```java
@Component
public class ResultEnrichInterceptor implements ToolCallbackInterceptor {

    @Override
    public void beforeCall(String toolName, String toolInput) {
        // no-op
    }

    @Override
    public String afterCall(String toolName, String toolInput, String toolOutput) {
        if ("queryUser".equals(toolName)) {
            return toolOutput + "\n注意：以上信息仅供内部使用";
        }
        return toolOutput;
    }
}
```

## 9. 写操作与风险控制

### 9.1 写操作确认机制

框架提供了写操作确认配置，当开启后，所有写操作（新增/修改/删除）会先展示给用户确认后才执行：

```yaml
ai:
  agent:
    safety:
      write-need-confirm: true   # 写操作需用户确认（推荐开启）
```

工作流程：

```
用户: "删除ID为5的用户"
  ↓
Agent 识别为写操作，生成确认提示
  ↓
Agent: "即将执行删除用户操作（ID=5），确认执行吗？"
  ↓
用户: "确认"
  ↓
Agent 调用 deleteUser(id=5) 执行删除
```

### 9.2 风险等级标识

`@AgentTool` 注解提供 `riskLevel` 属性，支持三个等级：

| 等级 | 枚举值 | 适用场景 | 建议 |
|------|--------|----------|------|
| 低风险 | `RiskLevel.LOW` | 只读查询 | 默认值，无需额外配置 |
| 中风险 | `RiskLevel.MEDIUM` | 数据新增/修改 | 建议开启确认 |
| 高风险 | `RiskLevel.HIGH` | 数据删除、批量操作 | 强烈建议开启确认 + 审计日志 |

使用示例：

```java
// 只读查询工具，低风险
@AgentTool(name = "report-tools", description = "报表查询工具", riskLevel = AgentTool.RiskLevel.LOW)
public class ReportTools { ... }

// 包含增删改的工具，中风险
@AgentTool(name = "user-tools", description = "用户管理工具", riskLevel = AgentTool.RiskLevel.MEDIUM)
public class UserTools { ... }

// 涉及批量删除或资金操作的工具，高风险
@AgentTool(name = "finance-tools", description = "财务操作工具", riskLevel = AgentTool.RiskLevel.HIGH)
public class FinanceTools { ... }
```

### 9.3 只读保护模式

Demo 模块中提供了只读保护开关，当开启时所有写操作需要二次确认：

```yaml
ai:
  agent:
    demo:
      enable-readonly-guard: true  # 只读保护
```

### 9.4 写操作的返回值惯例

写操作（新增/修改/删除）建议返回简短字符串（如 `"ok"`），AI 会据此生成用户友好的回复。不要返回操作的详细过程，LLM 只需要知道操作是否成功。

```java
@Tool(description = "新增用户")
public String addUser(UserDTO dto) {
    userService.add(convert(dto));
    return "ok";   // 简短明确
}

@Tool(description = "删除用户")
public String deleteUser(Long id) {
    userService.delete(id);
    return "ok";
}
```

### 10.3 工具类

```java
package com.example.tools;

import io.github.aiagent.core.tool.annotation.AgentTool;
import com.example.dto.EmployeeDTO;
import com.example.entity.Employee;
import com.example.service.EmployeeService;
import org.springframework.ai.tool.annotation.Tool;
import org.springframework.ai.tool.annotation.ToolParam;

import java.util.List;

@AgentTool(
    name = "employee-tools",
    description = "员工管理工具，支持查询、新增和修改员工信息",
    riskLevel = AgentTool.RiskLevel.MEDIUM
)
public class EmployeeTools {

    private final EmployeeService employeeService;

    public EmployeeTools(EmployeeService employeeService) {
        this.employeeService = employeeService;
    }

    /**
     * 查询操作：返回 List，框架自动格式化为表格。
     * 参数均可选，传 null 时忽略该条件。
     */
    @Tool(description = "按姓名/手机号/职位查询员工，参数均为可选，传空时查询全部")
    public List<Employee> queryEmployee(
            @ToolParam(description = "员工姓名，支持模糊匹配", required = false) String name,
            @ToolParam(description = "手机号", required = false) String phone,
            @ToolParam(description = "职位名称", required = false) String position) {
        return employeeService.query(name, phone, position);
    }

    /**
     * 新增操作：使用 DTO 接收参数，支持 @TranslateField 自动翻译。
     * 返回 "ok" 表示操作成功。
     */
    @Tool(description = "新增员工，需提供姓名、手机号等信息")
    public String addEmployee(EmployeeDTO dto) {
        Employee employee = convert(dto);
        employeeService.add(employee);
        return "ok";
    }

    /**
     * 修改操作：需提供员工 ID 和需要更新的字段。
     * 只传需要修改的字段即可，未传的字段不会被覆盖。
     */
    @Tool(description = "修改员工信息，需指定员工ID和要修改的字段")
    public String updateEmployee(
            @ToolParam(description = "员工ID", required = true) Long id,
            EmployeeDTO dto) {
        Employee employee = convert(dto);
        employee.setId(id);
        employeeService.update(employee);
        return "ok";
    }

    /**
     * 删除操作：高风险操作，配合 write-need-confirm 使用。
     */
    @Tool(description = "删除员工，需指定员工ID")
    public String deleteEmployee(
            @ToolParam(description = "员工ID", required = true) Long id) {
        employeeService.delete(id);
        return "ok";
    }

    private Employee convert(EmployeeDTO dto) {
        Employee employee = new Employee();
        employee.setName(dto.getName());
        employee.setPhone(dto.getPhone());
        employee.setEmail(dto.getEmail());
        employee.setPosition(dto.getPosition());
        employee.setDeptId(dto.getDeptId());
        employee.setStatus(dto.getStatus());
        return employee;
    }
}
```

## 12. 常见问题

### Q1: 工具未被 Agent 识别

**现象**：对话时 Agent 不调用你新增的工具，或 `/api/agent/tools` 中看不到该工具。

**排查步骤**：

1. **确认注解**：类上是否标注了 `@AgentTool`，方法上是否标注了 `@Tool`
2. **确认扫描路径**：工具类是否在 Spring Boot 的组件扫描路径下（默认扫描主类所在包及子包）
3. **确认白名单**：`@AgentTool(name = "xxx")` 中的 name 是否在 `tool-whitelist` 配置中
4. **确认依赖**：模块是否引入了 `ai-agent-core` 或 `ai-agent-spring-boot-starter`
5. **查看启动日志**：搜索 `AgentToolCallbackProvider` 相关日志，确认扫描到了几个 Bean

### Q2: 参数解析失败

**现象**：Agent 调用工具时报参数错误，或 LLM 传入的参数与方法签名不匹配。

**排查步骤**：

1. **检查参数名**：确认方法参数名有语义，且编译时保留了参数名（`-parameters` 编译选项或使用 `@ToolParam`）
2. **检查参数类型**：LLM 传入的是 JSON 字符串，复杂类型需要能被 Jackson 反序列化
3. **检查 DTO**：DTO 必须有无参构造函数和标准的 getter/setter
4. **检查 description**：`@Tool` 的 description 是否清晰描述了参数含义，LLM 依赖此信息填充参数

### Q3: 工具调用超时

**现象**：Agent 报错 `Tool execution failed after retries`。

**解决方案**：

1. **增大超时**：调整 `ai.agent.tool.timeout-seconds`，默认 10 秒
2. **优化查询**：检查工具方法中的 SQL 或远程调用是否耗时过长
3. **增加重试**：对网络相关的瞬态错误，适当增加 `ai.agent.tool.max-retries`
4. **检查索引**：数据库查询是否缺少索引导致全表扫描

### Q4: 并发限流被触发

**现象**：报错 `Tool concurrency limit reached`。

**解决方案**：

1. **增大并发数**：调整 `ai.agent.tool.max-concurrent`，默认为 5
2. **排查阻塞**：检查是否有工具方法执行时间过长，占用了并发槽位
3. **考虑分流**：如果是高并发场景，考虑优化工具执行效率而非无限增大并发数

### Q5: 返回结果格式混乱

**现象**：LLM 收到的结果不易理解，导致回复质量差。

**解决方案**：

1. **使用结构化返回**：查询操作返回 `List<Entity>` 而非手工拼接字符串
2. **控制返回量**：大列表做分页或限制条数（如 `previewRecordLimit`）
3. **精简字段**：使用 VO 而非 Entity，只返回必要字段
4. **检查序列化**：确保返回对象能被 Jackson 正确序列化

### Q6: 字段翻译不生效

**现象**：DTO 中的 `@TranslateField` 没有完成名称到 ID 的转换。

**排查步骤**：

1. 确认引入了 `ai-agent-translator` 模块
2. 确认 `@TranslateField` 的 `source` 与 LLM 传入的 JSON key 匹配
3. 确认字典表或目标表中存在对应数据
4. 查看日志中是否有翻译相关的错误信息

## 13. Checklist

新工具上线前，请逐项检查：

### 注解与注册

- [ ] 工具类标注了 `@AgentTool`，且 `name` 全局唯一
- [ ] 每个工具方法标注了 `@Tool`，且 `description` 清晰准确
- [ ] 工具名称已加入 `tool-whitelist` 配置
- [ ] 工具类在 Spring Boot 组件扫描路径下

### 参数设计

- [ ] 参数命名具有语义（LLM 能理解）
- [ ] 复杂写操作使用 DTO，简单查询使用松散参数
- [ ] 可选参数允许为 null，且方法内正确处理 null 值
- [ ] 需要字段翻译的 DTO 已标注 `@TranslateField`

### 返回值设计

- [ ] 查询操作返回 `List` / `VO` 等结构化类型
- [ ] 写操作返回简短字符串（如 `"ok"`）
- [ ] 返回对象可被 Jackson 正确序列化（有 getter、无循环引用）
- [ ] 大列表做了条数限制，避免 Token 浪费

### 安全与治理

- [ ] `riskLevel` 根据操作危险程度正确设置
- [ ] 写操作确认 `write-need-confirm` 已按需开启
- [ ] 工具方法不会返回敏感信息（密码、密钥等）
- [ ] 工具方法内做了必要的参数校验和异常处理

### 验证

- [ ] 应用启动后 `GET /api/agent/tools` 能看到新工具
- [ ] 通过对话测试，Agent 能正确识别并调用新工具
- [ ] 工具方法的成功和失败分支都已测试
- [ ] 超时、重试、并发限流等治理能力已确认生效
