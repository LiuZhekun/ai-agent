# 集成指南（后端开发者执行版）

## 1. 适用范围

本手册面向 Java/Spring Boot 后端团队，目标是将 `ai-agent` 能力按业务成熟度分级接入现有系统，并可在生产环境稳定运行。

适用对象：

- 使用 Spring Boot 3.x、JDK 17+ 的后端服务
- 需要提供统一 AI 对话 API、工具调用、知识库检索或流式返回能力的业务系统
- 需要分阶段上线（最简试点 -> 标准落地 -> 完整建设）的团队

不适用对象：

- 非 JVM 技术栈且暂无网关转发层的系统
- 未具备基础日志、配置中心、CI/CD 流程的环境

---

## 2. 集成前检查清单

在进入三级接入前，请先完成以下检查。建议按顺序执行。

### 2.1 环境与依赖

1. 检查 JDK 与 Maven 版本。

```bash
java -version
mvn -v
```

2. 确认 Spring Boot 主版本为 3.x（建议 3.2+）。
3. 确认网络可访问目标 LLM Provider（内网代理、白名单、证书链）。
4. 确认应用已有统一日志框架（Logback/Log4j2）和可观测链路（至少含 traceId）。

### 2.2 配置与密钥

1. 在配置中心或环境变量准备以下参数：
   - `AI_AGENT_ENABLED`
   - `AI_LLM_PROVIDER`
   - `AI_LLM_BASE_URL`
   - `AI_LLM_API_KEY`
2. 明确密钥管理策略（禁止明文提交到 Git 仓库）。
3. 准备不同环境配置：`dev`、`test`、`prod`。

### 2.3 运行与容量

1. 评估并发与超时预算（建议先给 AI 调用单独线程池与超时）。
2. 明确熔断/限流策略（网关限流、服务级限流至少配置其一）。
3. 预留回滚开关（例如 `ai.agent.enabled=false`）。

---

## 3. 三级接入方案

建议从最简接入起步，完成线上验证后再升级到标准与完整接入。

## 3.1 最简接入（快速可用）

### 3.1.1 目标

在最小改动下打通 AI 请求链路，实现可调用、可观测、可回滚。

### 3.1.2 改动点

- 引入 `ai-agent-spring-boot-starter` 依赖
- 增加基础 LLM 配置
- 暴露统一对话接口（如 `/api/agent/chat`）
- 增加最小日志埋点（请求耗时、状态码、traceId）

### 3.1.3 配置示例

`application.yml`：

```yaml
ai:
  agent:
    enabled: true
  llm:
    provider: openai-compatible
    base-url: https://your-llm-endpoint/v1
    api-key: ${AI_LLM_API_KEY}
    model: gpt-4o-mini
    connect-timeout-ms: 3000
    read-timeout-ms: 30000
```

Maven 依赖示例：

```xml
<dependency>
  <groupId>com.your-org</groupId>
  <artifactId>ai-agent-spring-boot-starter</artifactId>
  <version>${ai-agent.version}</version>
</dependency>
```

### 3.1.4 代码接入步骤（按步骤编号）

1. 拉取依赖并编译。

```bash
mvn -U clean package -DskipTests
```

2. 在业务服务配置文件新增 AI 基础配置（见 3.1.3）。
3. 确保控制器或网关路由可访问 `/api/agent/chat`。
4. 在调用入口补充日志字段：`traceId`、`userId`、`requestId`、`latencyMs`。
5. 重启服务并确认启动日志出现 AI 自动装配成功信息。

### 3.1.5 验证步骤

1. 健康检查。

```bash
curl -s http://127.0.0.1:8080/actuator/health
```

2. 最小对话调用。

```bash
curl -X POST "http://127.0.0.1:8080/api/agent/chat" ^
  -H "Content-Type: application/json" ^
  -d "{\"sessionId\":\"demo-001\",\"message\":\"请用一句话介绍你自己\"}"
```

3. 日志核对（Linux）。

```bash
tail -n 200 logs/app.log | grep -E "agent|llm|traceId"
```

4. 日志核对（PowerShell）。

```powershell
Get-Content .\logs\app.log -Tail 200 | Select-String "agent|llm|traceId"
```

### 3.1.6 回滚步骤

1. 将 `ai.agent.enabled` 改为 `false`。
2. 灰度发布并观察 5-10 分钟错误率与延迟。
3. 如仍异常，回退到上一个稳定制品版本。
4. 保留调用与错误日志，进入故障排查流程（见第 6 章）。

## 3.2 标准接入（可落地运营）

### 3.2.1 目标

支持业务工具调用、会话记忆、基础策略控制，满足生产可运营要求。

### 3.2.2 改动点

- 接入 `@AgentTool` 业务工具函数（查询订单、查询库存等）
- 启用记忆、澄清、规划能力开关
- 增加审计日志与错误码映射
- 增加限流、超时、重试策略

### 3.2.3 配置示例

`application.yml`：

```yaml
ai:
  agent:
    enabled: true
    features:
      memory: true
      planner: true
      clarify: true
    tool:
      timeout-ms: 2000
      max-calls-per-request: 5
  guardrail:
    enabled: true
    blocked-keywords:
      - 内部密钥
      - 原始身份证号
```

### 3.2.4 代码接入步骤（按步骤编号）

1. 定义工具类并添加注解（示例：订单查询工具）。
2. 为工具类补充参数校验与超时保护。
3. 将工具执行结果映射成可解释结构（避免把底层异常直接暴露给模型）。
4. 在会话层接入记忆存储（Redis/DB 任一）。
5. 在 API 层增加审计日志：输入摘要、工具调用次数、token 消耗、响应耗时。
6. 编写集成测试：工具成功、工具超时、参数非法三类用例。

参考命令：

```bash
mvn -Dtest=*Agent*Test test
```

### 3.2.5 验证步骤

1. 构造触发工具调用的请求（如“查询订单 12345 的状态”）。
2. 检查响应中是否包含工具结果融合后的自然语言输出。
3. 检查日志中是否记录工具调用链与耗时。
4. 压测 5 分钟观察错误率。

```bash
curl -X POST "http://127.0.0.1:8080/api/agent/chat" ^
  -H "Content-Type: application/json" ^
  -d "{\"sessionId\":\"tool-001\",\"message\":\"查询订单12345状态\"}"
```

```bash
curl -s "http://127.0.0.1:8080/actuator/metrics/http.server.requests"
```

### 3.2.6 回滚步骤

1. 关闭高风险能力开关（优先关闭 `planner`、`memory`）。
2. 将工具调用切回兜底文案（仅返回人工客服引导或静态答案）。
3. 保留最简接入链路继续服务，避免完全不可用。
4. 若仍异常，整体回退至最简接入发布版本。

## 3.3 完整接入（企业级能力）

### 3.3.1 目标

接入知识库检索（L3 RAG）、向量化、翻译、多通道协议与全链路观测，实现可扩展的企业级 AI 平台能力。

### 3.3.2 改动点

- 接入 knowledge 与分层检索（关键词召回 + 向量召回 + 重排）
- 接入 vectorizer、translator
- 前后端通过 SSE 事件协议联调
- 增加指标看板与告警规则（延迟、错误率、命中率、token 成本）

### 3.3.3 配置示例

`application.yml`：

```yaml
ai:
  rag:
    enabled: true
    level: L3
    top-k: 8
    rerank: true
  vectorizer:
    provider: bge
    dimensions: 1024
  translator:
    enabled: true
    source-lang: auto
    target-lang: zh-CN
  sse:
    enabled: true
    heartbeat-seconds: 15
```

### 3.3.4 代码接入步骤（按步骤编号）

1. 建立知识文档入库流程（清洗 -> 分片 -> 向量化 -> 建索引）。
2. 配置 RAG 检索链路与重排策略，控制 `topK` 与阈值。
3. 对接 translator，在入口/出口增加语言识别与翻译策略。
4. 开启 SSE 输出，在网关与前端约定事件类型（`message`、`tool`、`done`、`error`）。
5. 增加链路指标上报（Prometheus/Micrometer）。
6. 增加告警规则（5xx 比例、P95 延迟、RAG 命中率）。
7. 完成全链路联调并出具压测报告。

示例命令：

```bash
mvn -pl ai-agent-knowledge -am clean test
```

```bash
curl -N "http://127.0.0.1:8080/api/agent/chat/stream?sessionId=sse-001&message=介绍一下退款流程"
```

### 3.3.5 验证步骤

1. 验证知识召回：输入领域问题，检查返回内容是否引用知识库语义。
2. 验证 SSE：检查事件顺序和结束事件是否完整。
3. 验证 translator：输入英文问题，输出中文答案（或按业务配置）。
4. 验证资源与成本：检查 token 消耗、QPS、内存、CPU。
5. 进行故障注入：模拟向量库不可用、LLM 超时、翻译服务失败，确认降级策略生效。

### 3.3.6 回滚步骤

1. 先关闭 `rag.enabled`、`translator.enabled`，保留核心对话能力。
2. 将接口从 SSE 降级为普通同步响应（必要时）。
3. 暂停知识入库任务，避免错误索引继续扩散。
4. 回退配置与制品，恢复到标准接入版本。

## 4. 版本与兼容性建议

### 4.1 推荐版本基线

- JDK：17 或 21（长期支持版本）
- Spring Boot：3.2.x 或 3.3.x
- Maven：3.9+
- Redis：6.x/7.x（用于记忆与缓存）
- 向量数据库：选择团队已运维的稳定方案（避免一次性引入过多新组件）

### 4.2 兼容性策略

1. 使用 BOM 或父 POM 锁定关键依赖版本，避免传递依赖漂移。
2. 业务系统升级 `ai-agent` 版本时，先在 `test` 环境做回归：
   - 核心接口兼容性
   - 工具调用行为
   - RAG 召回质量
3. 对外 API 建议遵循“新增字段不破坏旧客户端”的向后兼容原则。

## 6. 常见故障与排查命令

### 6.1 启动后接口 404

可能原因：

- 自动装配未生效
- 路由前缀配置错误

排查命令：

```bash
curl -s http://127.0.0.1:8080/actuator/mappings
```

```bash
tail -n 200 logs/app.log | grep -E "AutoConfiguration|agent|mapping"
```

### 6.2 接口超时或响应极慢

可能原因：

- LLM 上游延迟高
- 线程池耗尽
- 工具调用阻塞

排查命令：

```bash
curl -s http://127.0.0.1:8080/actuator/metrics/http.server.requests
```

```bash
curl -s http://127.0.0.1:8080/actuator/threaddump
```

### 6.3 工具调用不生效

可能原因：

- `@AgentTool` 未被扫描
- 能力开关关闭
- 参数校验失败被拦截

排查命令：

```bash
tail -n 300 logs/app.log | grep -E "tool|AgentTool|validation|timeout"
```

```bash
curl -s http://127.0.0.1:8080/actuator/configprops
```

### 6.4 SSE 中断或前端收不到结束事件

可能原因：

- 网关超时配置过短
- 心跳未开启
- 客户端断流未重连

排查命令：

```bash
curl -N "http://127.0.0.1:8080/api/agent/chat/stream?sessionId=check-sse&message=ping"
```

```bash
tail -n 300 logs/app.log | grep -E "sse|heartbeat|disconnect|done"
```

## 7. 落地建议

建议采用“1 周最简、2-3 周标准、按业务节奏推进完整”的路线，先保证可用与可观测，再逐步引入高阶能力。每次升级档位都必须保留上一档位的回滚开关，确保线上稳定性优先。
